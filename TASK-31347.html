Issue 1 — Customer Not Fetching on Debit/Credit Note Page

Customer: INTERIOR THREE SIXTY LLC  |  Branch / Invoice / Job: not supplied in ticket (CN request attached as a file only)

Key Takeaway The Debit/Credit Note customer autocomplete is gated on the ops customer's active flag (masterOps.Customer.isActive = 1), not on the finance ledger. A customer that has a perfectly healthy, active ledger and open invoices can still vanish from the CN screen if the ops record has been deactivated. When a customer "won't fetch" for a CN/receipt, check masterOps.Customer.isActive first, before looking at the ledger side.
Problem

On Invoice > Invoice > Debit/Credit Note, the user could not raise a Credit Note for customer INTERIOR THREE SIXTY LLC. Typing the customer name into the Customer Name field returned no match, so the form could not resolve the customer and the CN could not be created.

No job number, invoice number, or branch was supplied in the ticket, the CN request was attached as a file rather than pasted, so the starting point was the customer name alone.

Investigation
Step 1 — Identify what feeds the CN "Customer Name" field

In the CN component (credit-note.component.ts) the customer autocomplete is wired to CustomerNameAutoCompleteUrl = "/Receipt/Get_Customer_Debtor", which the BSSApi (FIN_ReceiptController) maps to stored procedure Fin.Get_Customer_Debtor (@Search, @Branch). Reading the SP body: 

select l.LedgerName+' - '+isnull(mc.CuPrintName,'')+' - '+isnull(mc.BranchCode,'') Label,
       l.LedgerID Value, fc.CustomerID, l.LedgerID CustomerLedgerID
from fin.LedgerMaster l
left join fin.FCustomer fc        on fc.LedgerID = l.LedgerID
left join masterops.Customer mc   on mc.CustomerId = fc.OpsID
left join ff.Branch br            on br.BranchCode = mc.BranchCode
where l.ParentID = 27                        -- SUNDRY DEBTORS
  and l.LedgerName like '%'+@Search+'%'       -- searches LEDGER name, not CuName
  and mc.isActive = 1 and fc.isActive = 1     -- both ops + finance must be active

Two things stand out: the search is on LedgerMaster.LedgerName (not the ops CuName), and a row is dropped unless the ledger sits under ParentID = 27 (Sundry Debtors) and both mc.isActive = 1 and fc.isActive = 1. The @Branch / company filter is commented out, so branch is not what excludes a row here. The candidate reasons the customer is missing are therefore: misnamed/absent ledger, wrong ParentID, or one of the two active flags being 0.

Step 2 — Trace the customer through the whole chain with no filters

Starting from the ops customer and following the exact joins the SP uses, but without any of the WHERE filters, so it is visible which single condition drops the row. A parallel search on the ledger side rules out a name-spelling mismatch (the field the autocomplete actually matches on). 

-- 1: trace INTERIOR THREE SIXTY LLC through the Get_Customer_Debtor chain
SELECT  mc.CustomerId, mc.CuName, mc.CuPrintName, mc.BranchCode AS Cust_BranchCode,
        mc.isActive AS Cust_isActive,
        fc.CustomerID AS FCustomerID, fc.OpsID, fc.LedgerID, fc.isActive AS FCust_isActive,
        l.LedgerID AS Ledger_LedgerID, l.LedgerName, l.ParentID AS Ledger_ParentID, l.isActive AS Ledger_isActive
FROM masterOps.Customer mc
LEFT JOIN fin.FCustomer   fc ON fc.OpsID   = mc.CustomerId
LEFT JOIN fin.LedgerMaster l ON l.LedgerID = fc.LedgerID
WHERE mc.CuName LIKE '%INTERIOR THREE SIXTY%'
   OR mc.CuPrintName LIKE '%INTERIOR THREE SIXTY%';

-- 1b: does any SUNDRY DEBTOR ledger name contain the string the user types?
SELECT l.LedgerID, l.LedgerName, l.ParentID, l.isActive
FROM fin.LedgerMaster l
WHERE l.LedgerName LIKE '%INTERIOR%';
Query 1 (one row): CustomerId = 50029 | CuName = INTERIOR THREE SIXTY LLC | CuPrintName = INTERIOR THREE BranchCode = Y | Cust_isActive = 0 <-- INACTIVE FCustomerID = 11503 | OpsID = 50029 | LedgerID = 16492 | FCust_isActive = 1 Ledger_LedgerID = 16492 | LedgerName = INTERIOR THREE SIXTY LLC | ParentID = 27 | Ledger_isActive = 1 Query 1b (ledger 16492 present, correct name/parent/active): 16492 INTERIOR THREE SIXTY LLC 27 1 (only INTERIOR THREE SIXTY ledger; others are unrelated names)

The entire chain is intact: the ledger exists (16492) under ParentID = 27 and is active, the FCustomer (11503) links it to the ops customer and is active, and the ledger name matches the search string exactly. The only SP condition that fails is mc.isActive = 1, the ops customer row (CustomerId 50029, branch Y) has isActive = 0. That single flag is what excludes it from the autocomplete.

Step 3 — Confirm the deactivation context and the invoice the CN targets

Before touching the flag, capture when/who deactivated the customer, and confirm there is a genuine open invoice the CN is meant to credit (which also recovers the invoice number the ticket never supplied). 

-- 3a: deactivation audit
SELECT CustomerId, CuName, CuPrintName, BranchCode, isActive,
       CreatedBy, Created, UpdatedBy, Updated
FROM masterOps.Customer WHERE CustomerId = 50029;

-- 3b: outstanding sales invoices for this customer (FCustomerID 11503)
SELECT ii.InvoiceID, i.InvoiceNumber, i.InvoiceDate, i.Branch, ii.FCustomerID,
       round(ii.InvAmount,0) InvAmount, round(ii.Received,0) Received, isNull(ii.CreditAmt,0) CreditAmt,
       round(ii.InvAmount,0)-round(ii.Received,0)-round(isNull(ii.CreditAmt,0),0) Outstanding, ii.isActive
FROM ff.Invoice_Inventory ii
LEFT JOIN ff.Invoice i ON i.InvoiceId = ii.InvoiceID
WHERE ii.FCustomerID IN (11503, 50029)
ORDER BY i.InvoiceDate DESC;
3a: CustomerId 50029 Created 2026-03-24 by user 2415 | Updated (deactivated, isActive=0) 2026-03-25 13:59 by user 111 3b: InvoiceID 228962 | InvoiceNumber 26700996 | InvoiceDate 2026-03-23 | Branch Y | FCustomerID 11503 InvAmount 24,609.00 | Received 0.00 | CreditAmt 0.00 | Outstanding 24,609.00 | isActive 1

The customer was created on 24 Mar 2026 and deactivated the very next day (25 Mar 2026) by user 111. There is exactly one open invoice, 26700996, fully outstanding at 24,609.00, tied to this customer's FCustomerID (11503), a legitimate reason to raise the CN. Step 2 already established there is no other ops customer or ledger named "INTERIOR THREE SIXTY", so this is not a surviving duplicate holding the invoice; 50029 / 11503 is the single record for this customer.

Root Cause
masterOps.Customer, CustomerId 50029 (INTERIOR THREE SIXTY LLC, branch Y) — isActive = 0 The customer is deactivated on the ops side. Fin.Get_Customer_Debtor, which powers the Debit/Credit Note "Customer Name" autocomplete, requires mc.isActive = 1, so the deactivated customer never appears and the CN form cannot resolve it. The finance ledger (16492, ParentID 27, active) and FCustomer (11503, active) are fully intact, the ops active flag is the only blocker.
Confirmation Queries

The row that proved the root cause, filtered to the failing condition alone:

SELECT CustomerId, CuName, BranchCode, isActive
FROM masterOps.Customer
WHERE CustomerId = 50029;
-- returns: 50029 | INTERIOR THREE SIXTY LLC | Y | 0   (isActive = 0)

Re-running Fin.Get_Customer_Debtor with @Search = 'INTERIOR' returns the customer only after isActive is set back to 1.

Solution

Reactivate the ops customer so Fin.Get_Customer_Debtor returns it. Single row, fully reversible. Preview first, apply, then verify on the CN screen.

Status at time of writing The fix below is documented as to apply and verify, it had not yet been run against the database or confirmed on the Debit/Credit Note screen when this report was written. Once the UPDATE is applied and the customer is confirmed to resolve on the CN page with invoice 26700996 selectable, this note should be updated to reflect the fix as applied & verified. 
-- Preview (before) — expect 1 row, isActive = 0
SELECT CustomerId, CuName, BranchCode, isActive, UpdatedBy, Updated
FROM masterOps.Customer
WHERE CustomerId = 50029;

-- Fix — expect 1 row affected
UPDATE masterOps.Customer
SET isActive = 1
WHERE CustomerId = 50029;   -- reactivate INTERIOR THREE SIXTY LLC (branch Y)

Verify in the frontend: Invoice > Invoice > Debit/Credit Note (DrCr = Credit) → type "INTERIOR" in Customer Name → "INTERIOR THREE SIXTY LLC - INTERIOR THREE - Y" appears → select it, then in the grid's Credit Invoice No search 26700996 → it lists with 24,609 outstanding, confirming the CN can be raised end to end.

Revert query (if the reactivation needs to be undone) 
UPDATE masterOps.Customer
SET isActive = 0
WHERE CustomerId = 50029;
Confirm before applying The deactivation was a deliberate action by user 111 on 25 Mar 2026, one day after the customer was created, a pattern that can indicate an intended duplicate cleanup. Since there is a live open invoice (26700996) requiring a CN and no surviving duplicate was found, reactivation is the correct fix, but it is worth a quick confirmation with the owner of user 111 that the deactivation was accidental / that reactivating for CN purposes is acceptable.
