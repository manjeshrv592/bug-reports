Issue 1 — Invoice TN270400263 not on Cheque Request Page

Job: 2605924  |  Branch: Rotterdam (R)  |  ICostID: 312804

Key Takeaway If TallyPosted is set without ChqReqSent being set first, the invoice is permanently hidden from the Cheque Request page. This is an invalid state — always check both columns together when an invoice is missing.
Problem

Invoice TN270400263 for Job 2605924 (Rotterdam) was entered on the Buy Rate page but was not appearing on the Cheque Request page, so it could not be sent for payment.

The Cheque Request page calls ff.Get_2ChequeRequest @Branch='R'. The invoice was not in the result set.

Investigation
Step 1 — Identify which job type the invoice belongs to

The SP handles multiple job types: Export, Import, Misc, CHA. Ran a union query across all tables to locate the invoice.

SELECT 'Export' AS Type, c.VendorInvNo, j.JobNumber
FROM ff.BB_Cost c JOIN ff.BJob j ON j.JobID = c.JobID
WHERE j.JobNumber = '2605924'
UNION ALL
SELECT 'Import', ic.VendorInvNo, bi.BookingNo
FROM ff.ICost ic JOIN ff.BBooking_Import bi ON bi.BookingId = ic.BookingID
WHERE bi.BookingNo = '2605924'
UNION ALL
SELECT 'Misc', ic.VendorInvNo, mj.JobNo
FROM ff.MJ_Cost ic JOIN ff.MiscJob mj ON mj.JobId = ic.JobID
WHERE mj.JobNo = '2605924'
UNION ALL
SELECT 'CHA', ic.VendorInvNo, cj.JobNo
FROM ff.CC_Cost ic JOIN Cha.Job_CHA cj ON cj.CJobID = ic.JobID
WHERE cj.JobNo = '2605924'
Import TN270400263 2605924

Invoice belongs to the Import section — table ff.ICost.

Step 2 — Check all SP filter conditions for this record

The Import section of the SP filters on several columns. Retrieved all relevant values for this invoice.

SELECT

Root Cause
ff.ICost — ICostID 312804: TallyPosted = 1 The SP filter AND ic.TallyPosted IS NULL was excluding this record. The field TallyPosted was set to 1, but ChqReqSent is still NULL — meaning this entry was marked as Tally-posted without ever going through the Cheque Request step. This is not a valid state and was confirmed to be set by mistake.
Solution

Finance team confirmed TallyPosted was set by mistake — the invoice had not been posted in Tally. Fix applied on 30 June 2026:

UPDATE ff.ICost
SET TallyPosted = NULL
WHERE ICostID = 312804

After this update, the invoice appeared on the Cheque Request page and was processed normally.

Issue 2 — Invoice 26700232 not on Cheque Request Page

Job: ANTI20240294  |  Branch: Antwerp (W)  |  BookingID: 41674  |  ICostIDs: 335395, 335396

Key Takeaway The SP filters imports by BookingDate, not invoice date. An invoice added to an old booking will be cut off even if the invoice itself is recent. Fix: extend the SP filter to also allow records where VendorInvDate >= cutoff.
Problem

Invoice 26700232 for Job ANTI20240294 (Antwerp) was not appearing on the Cheque Request page. The invoice had 2 cost lines (ICostID 335395 and 335396) entered recently in 2026, but the booking itself dated back to 2024.

Same SP: ff.Get_2ChequeRequest @Branch='W'. Invoice was absent from the result set in the frontend, but when we ran the SP directly in SQL it appeared — pointing to a date filter in the SP that was context-dependent.

Investigation
Step 1 — Confirm the invoice exists and check its ICost records
SELECT
    ic.ICostID, ic.VendorInvNo, ic.isActive,
    ic.ChqReqSent, ic.TallyPosted, ic.CRevenueID,
    bi.BookingNo, bi.Branch, bi.BookingDate
FROM ff.ICost ic
JOIN ff.BBooking_Import bi ON bi.BookingId = ic.BookingID
WHERE ic.VendorInvNo = '26700232'
ICostID VendorInvNo isActive ChqReqSent TallyPosted CRevenueID BookingNo Branch BookingDate 335395 26700232 1 NULL NULL NULL ANTI20240294 W 2024-01-02 335396 26700232 1 NULL NULL NULL ANTI20240294 W 2024-01-02

All SP filters pass except one — the BookingDate is 2024-01-02.

Step 2 — Identify the date filter in the SP

The Import section of ff.Get_2ChequeRequest contains this date cutoff:

AND bi.BookingDate >= CASE WHEN bi.Branch = 'Y' THEN '2026-01-01' ELSE '2025-01-01' END

Branch W (Antwerp) is not 'Y', so it requires BookingDate >= '2025-01-01'. This booking's date is 2024-01-02, which fails the filter even though the invoice itself was created in 2026.

Step 3 — Why SP showed it in SQL but not in frontend

During investigation we temporarily changed the SP filter cutoff to '2024-01-01'. Running EXEC ff.Get_2ChequeRequest @Branch='W' in SQL did return invoice 26700232. However, the frontend still did not show it — this was a browser cache issue. Testing in Postman confirmed the API was returning the correct data once the date was changed. After reverting the SP to '2025-01-01', the invoice disappeared again — confirming the root cause.

Root Cause
ff.BBooking_Import — BookingNo ANTI20240294: BookingDate = 2024-01-02 The SP filter AND bi.BookingDate >= '2025-01-01' excludes this booking because it was created in 2024. The invoice (26700232) was added to this booking in 2026, but the SP uses the booking's original date — not the invoice date — for the cutoff check.
Solution

Temporary fix applied — 30 June 2026:

Changed the booking's BookingDate to today so it passes the SP filter. Must be reverted after the cheque request is processed. 

-- Original value recorded before change:
-- BookingID: 41674 | BookingNo: ANTI20240294 | BookingDate: 2024-01-02

-- Applied:
UPDATE ff.BBooking_Import
SET BookingDate = GETDATE()
WHERE BookingNo = 'ANTI20240294'

-- REVERT after cheque request is processed:
UPDATE ff.BBooking_Import
SET BookingDate = '2024-01-02'
WHERE BookingNo = 'ANTI20240294'
Revert required Once the cheque request for ANTI20240294 is sent and processed, run the revert query above to restore the original BookingDate.

Permanent fix (pending — needs team discussion):

Update the Import section of ff.Get_2ChequeRequest to also allow old bookings that have recent invoices:

-- Replace:
AND bi.BookingDate >= CASE WHEN bi.Branch = 'Y' THEN '2026-01-01' ELSE '2025-01-01' END

-- With:
AND (
    bi.BookingDate >= CASE WHEN bi.Branch = 'Y' THEN '2026-01-01' ELSE '2025-01-01' END
    OR ic.VendorInvDate >= '2025-01-01'
)
Note This permanent fix should be applied by the dev team and tested in sandbox before going to production. It ensures that even if a booking is old, any invoice with a date after the cutoff will still appear on the Cheque Request page.
