Issue 1 — Invoices 2604421 / 2604777 / 2604789 not on Purchase Posting Approval page

Jobs: 2604421 / 2604777 / 2604789  |  Page: Purchase Posting Approval

Key Takeaway The Purchase Posting Approval page only shows cheque requests sent within the last 2 months (ChqReqSent >= DATEADD(MONTH, -2, GETDATE())). A request that sits unapproved for more than 2 months silently disappears from the approval screen. Also: the branch in a captured API request is the logged-in user's branch, not necessarily the branch the missing jobs belong to — always verify which branch the jobs actually belong to before filtering.
Problem

Bug report received: “Purchase Posting not showing Invoice Number 2604421, 2604777 & 2604789.” On the Purchase Posting Approval page, the table of pending invoices did not contain these three entries.

The page calls the API endpoint ChequeRequest/Get_2ChequeRequest_Approve. A request captured during triage (Branch=T&DCase=0&VendorID=0) returned 200 OK with a populated result set, but the three invoices were not in it.

Investigation
Step 1 — Identify the stored procedure and its filters

From the backend controller BSSApi\Controllers\FIN_ChequeRequestController.cs, the endpoint calls SP ff.Get_2ChequeRequest_Approve, passing the URL's Branch value as @BranchList (comma-separated list) plus @DCase and @VendorID.

With DCase=0 and VendorID=0 the date window is wide open and there is no vendor filter. The job numbers match the Export pattern, so the relevant SP block reads ff.BB_Cost with these conditions: 

where isNull(c.VendorInvNo,'')<>''
  and isNull(c.TallyPosted,0)=0
  and c.JobID>0
  and isNull(c.ChqReqBy,0)<>0
  and c.DueDate between @D1 and @D2
  and c.isActive=1
  and c.ChqReqSent >= DATEADD(MONTH, -2, GETDATE())

Branch is applied via inner join @tblBranch tb on tb.Branch = bo.EdiCode, where bo comes from BBooking.BilledBy → masterCom.OrganizationUnit — i.e. the billing branch of the job's booking, not the branch in the URL.

Step 2 — Check every SP filter on the cost rows of the three jobs
SELECT j.JobNumber, c.CostID, c.JobID, c.PartyID,
       c.VendorInvNo, c.PurEntryNo, c.DueDate,
       c.TallyPosted, c.isActive, c.ChqReqBy, c.ChqReqSent
FROM ff.BJob j
JOIN ff.BB_Cost c ON c.JobID = j.JobID
WHERE j.JobNumber IN ('2604421', '2604777', '2604789')
ORDER BY j.JobNumber, c.CostID;

30 rows returned. Each job has invoices from three vendors. Two vendors' invoices already have TallyPosted = 1 (correctly excluded — already posted). One pending row per job remained, matching exactly the “invoice numbers” from the bug report (the vendor invoice number equals the job number):

Step 3 — Verify which branch the jobs actually belong to

The PurEntryNo values start with VINVM, while the captured API response (Branch=T) contained only VINVT entries — suggesting the jobs belong to another branch. Checked the full branch chain the SP uses, without assuming anything: 

SELECT j.JobNumber, j.JobID,
       j.Branch AS JobBranch,
       bd.BookingID,
       bk.BilledBy,
       bo.EdiCode AS BilledByEdiCode
FROM ff.BJob j
LEFT JOIN ff.BJob_Detail bd ON bd.JobID = j.JobID
LEFT JOIN ff.BBooking bk    ON bk.BookingID = bd.BookingID
LEFT JOIN masterCom.OrganizationUnit bo ON bo.OuId = bk.BilledBy
WHERE j.JobNumber IN ('2604421', '2604777', '2604789')
ORDER BY j.JobNumber, bd.BookingID;
JobNumber JobID JobBranch BookingID BilledBy BilledByEdiCode 2604421 78699 M 84677 19 M 2604777 78959 M 84961 19 M 2604789 78966 M 84968 19 M

All three jobs belong to branch M (Mumbai) end to end — job branch, booking billing branch, everything. ff.Branch confirms M = Mumbai, T = Tuticorin. Each job has exactly one booking, so the SP's TOP 1 booking lookup is not a factor. The captured Branch=T request was simply from a Tuticorin-branch login and could never contain Mumbai jobs — a red herring. The real complaint is about the Mumbai view.

Step 4 — Run the SP for branch M to confirm the date filter is the only blocker
EXEC ff.Get_2ChequeRequest_Approve @BranchList = 'M', @DCase = 0, @VendorID = 0;

Result: 40+ branch-M rows returned, but jobs 2604421 / 2604777 / 2604789 were absent. Crucially, other invoices from the same vendor (PartyID 38725) did appear (e.g. jobs 2605761, 2606248, 2606249) — their cheque requests were sent more recently. This rules out the branch join, the vendor, and every other filter: the ChqReqSent 2-month cutoff is the sole cause.

Root Cause
ff.BB_Cost — CostIDs 361963, 364537, 364527: ChqReqSent = 2026-04-21, older than the SP's rolling 2-month window The SP filter AND c.ChqReqSent >= DATEADD(MONTH, -2, GETDATE()) only shows cheque requests sent within the last 2 months. These three requests were sent on 2026-04-21 and were still awaiting approval when the window moved past them (cutoff on 06 Jul 2026 = 2026-05-06). They aged out of the Purchase Posting Approval page while pending — 15 days too old — and vanished for all branches.
Solution

Fix verified — 06 July 2026:

Re-dated ChqReqSent to the current date on the three pending cost rows so they re-enter the SP's 2-month window. Original values recorded first via a preview SELECT. 

-- Preview (returned exactly 3 rows; original ChqReqSent = 2026-04-21 on all):
SELECT CostID, JobID, VendorInvNo, PurEntryNo, PartyID,
       ChqReqSent, ChqReqBy, TallyPosted, isActive
FROM ff.BB_Cost
WHERE CostID IN (361963, 364537, 364527);

-- Fix applied (3 rows affected):
UPDATE ff.BB_Cost
SET ChqReqSent = GETDATE()
WHERE CostID IN (361963, 364537, 364527)
  AND ISNULL(TallyPosted, 0) = 0;   -- safety: never touch a posted row

-- REVERT (restores original state):
UPDATE ff.BB_Cost
SET ChqReqSent = '2026-04-21'
WHERE CostID IN (361963, 364537, 364527);

After the update, all three invoices appeared in the SP output for @BranchList='M' and on the Purchase Posting Approval page in the frontend.

Fix was reverted after verification — database is back in its original state The fix was applied, verified in the frontend on 06 July 2026, and then reverted using the revert query above. The three invoices are therefore still hidden. To surface them for approval, re-run the fix UPDATE (expected: 3 rows affected). Note that re-dating ChqReqSent rewrites when the cheque request was nominally sent — inform Finance before applying.

Permanent fix (pending — needs team discussion):

The rolling 2-month window on ChqReqSent exists in the Export, Import, Misc and Transport sections of ff.Get_2ChequeRequest_Approve (the CHA section uses a fixed JobDate >= '2025-01-01' instead, per Task 29936). Any entry pending approval for more than 2 months disappears. Options to discuss with the dev/finance team: 

-- Option A: widen the window (e.g. 6 months)
and c.ChqReqSent >= DATEADD(MONTH, -6, GETDATE())

-- Option B: fixed cutoff date, matching the CHA section's approach
and c.ChqReqSent >= '2025-01-01'
Note Any permanent change must be applied consistently to all four affected sections of the SP and tested in sandbox first. The window presumably exists to keep the approval list short, so the trade-off between window size and list length is a business decision.
