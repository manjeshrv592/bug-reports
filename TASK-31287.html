Issue 1 — HBL No Showing Incorrect on Invoice Page

Job 2609977  |  Dubai (DXB)  |  MBL MEDUYF745606  |  Invoice > Invoice > Invoice

Key Takeaway When a BL number field displays something that reads like a ship name, check the stored record against masterOps.VesselMaster before suspecting the display SP. HBLNo is a free-text varchar(30) written verbatim from the booking screen — the screen accepts anything, and the Invoice page faithfully shows whatever was saved.
Problem

Dubai branch reported that for job 2609977 (MBL MEDUYF745606) the HBL No is showing as “MSC EMANUELA” on the Invoice page (Invoice > Invoice > Invoice). “MSC EMANUELA” is the name of an MSC container vessel, not a bill of lading number. The branch asked whether the problem is in a procedure or in the database.

Investigation
Step 1 — Locate the job and see what is actually stored

Job numbers are not unique across job types, so we searched both the import side (ff.BBooking_Import + ff.BBooking_Import_HBL) and the export side (ff.BBooking + ff.BBL) by job number and MBL number. The goal: determine whether the database itself holds “MSC EMANUELA” (data problem) or holds a proper BL number that the screen is mangling (display/SP problem). 

-- 1a: Import side
SELECT bk.BookingID, bk.BookingNo, bk.Branch, bk.Segment, bk.MBLNo,
       bk.isActive AS BkActive,
       h.HBLID, h.HBLNo, h.HBLDate, h.isActive AS HblActive
FROM ff.BBooking_Import bk
LEFT JOIN ff.BBooking_Import_HBL h ON h.BookingID = bk.BookingID
WHERE bk.BookingNo = '2609977' OR bk.MBLNo = 'MEDUYF745606';

-- 1b: Export side
SELECT b.BookingID, b.BookingNumber, b.Branch, b.Segment, b.isActive,
       bl.BLID, bl.BLNumber AS HBLNo, bl.MBLNo
FROM ff.BBooking b
LEFT JOIN ff.BBL bl ON bl.BookingID = b.BookingID
WHERE b.BookingNumber = '2609977' OR bl.MBLNo = 'MEDUYF745606';
1a: BookingID 86123 | BookingNo 2609977 | DXB | Segment I2 | MBLNo MEDUYF745606 | active HBLID 89323 | HBLNo = 'MSC EMANUELA' | HBLDate 2026-06-17 | active 1b: (no rows)

The job is a sea import booking at Dubai with a single HBL, and the HBL record itself has HBLNo = 'MSC EMANUELA' stored in it. This is a data problem, not a display or procedure problem — the Invoice page shows exactly what is saved.

Step 2 — Compare the stored HBL No against the booking's vessel

The HBL entry section of the Import Booking screen carries vessel fields (VesselID, Voyage) alongside the HBL No box, and ff.Update_BBooking_Import writes the screen payload to the table verbatim. If the booking's assigned vessel is MSC EMANUELA, the vessel name was entered into the HBL No field. We also checked whether the HBL table carries audit columns. 

-- 2a: HBL row with vessel + booking context
SELECT h.HBLID, h.HBLNo, h.HBLDate, h.VesselID, v.VesselName, h.Voyage,
       h.CustomerRefNo, h.SAPNo, h.ShippedOnBoard,
       bk.BookingDate, bk.MBLNo, bk.MBLDate, bk.NoOfHBL
FROM ff.BBooking_Import_HBL h
LEFT JOIN masterOps.VesselMaster v ON v.VesselID = h.VesselID
JOIN ff.BBooking_Import bk ON bk.BookingID = h.BookingID
WHERE h.HBLID = 89323;

-- 2b: audit columns on ff.BBooking_Import_HBL?
SELECT c.name
FROM sys.columns c

  AND (c.name LIKE '%Creat%' OR c.name LIKE '%Modif%' OR c.name LIKE '%Edit%' OR c.name LIKE '%User%');
2a: HBLID 89323 | HBLNo 'MSC EMANUELA' | HBLDate 2026-06-17 VesselID 5438 | VesselName = 'MSC EMANUELA' | Voyage QS619A CustomerRefNo (blank) | SAPNo (blank) | ShippedOnBoard 2026-05-14 BookingDate 2026-06-17 | MBLNo MEDUYF745606 | MBLDate 2026-06-17 | NoOfHBL 1 2b: (no rows — the HBL table has no audit columns)

Confirmed: the booking's vessel (VesselID 5438) is literally named MSC EMANUELA — the same text sitting in the HBL No field. The vessel name was entered into the HBL No box. CustomerRefNo and SAPNo are both blank, so the correct HBL number is not recoverable from the system — it must come from the branch / origin agent documents.

Step 3 — Check whether the same mistake exists on other jobs

Swept every active import HBL in the system for rows where the stored HBL No equals the name of the vessel assigned to that HBL. Also listed audit columns on the booking header to identify who made the entry. 

-- 3a: audit columns on the booking header
SELECT c.name
FROM sys.columns c
WHERE c.object_id = OBJECT_ID('ff.BBooking_Import')
  AND (c.name LIKE '%Creat%' OR c.name LIKE '%Modif%' OR c.name LIKE '%Copie%' OR c.name LIKE '%User%');

-- 3b: recurrence sweep across all active import HBLs
SELECT bk.Branch, bk.BookingNo, bk.BookingDate, h.HBLID, h.HBLNo, v.VesselName
FROM ff.BBooking_Import_HBL h
JOIN ff.BBooking_Import bk ON bk.BookingID = h.BookingID
JOIN masterOps.VesselMaster v ON v.VesselID = h.VesselID
WHERE h.isActive = 1 AND bk.isActive = 1
  AND UPPER(LTRIM(RTRIM(h.HBLNo))) = UPPER(LTRIM(RTRIM(v.VesselName)))
ORDER BY bk.BookingDate DESC;

Only one other match in ~70,000 import HBLs: a 2023 Rotterdam job where the “vessel” in VesselMaster is itself named 7743 3118 8591 (a master-data oddity, not the same entry mistake). This is an isolated incident, not a pattern.

Step 4 — Identify who created the booking and when
SELECT bk.BookingID, bk.BookingNo, bk.CreatedBy, u.UserName AS CreatedByName, bk.Created,
       bk.CopiedFromID, bk.CopiedBy, u2.UserName AS CopiedByName, bk.Copied
FROM ff.BBooking_Import bk
LEFT JOIN masterUser.UserMaster u  ON u.UserID = bk.CreatedBy
LEFT JOIN masterUser.UserMaster u2 ON u2.UserID = bk.CopiedBy
WHERE bk.BookingID = 86123;
BookingID 86123 | BookingNo 2609977 CreatedBy 2752 (David Livingston) | Created 2026-06-17 19:19:10 CopiedFromID NULL | CopiedBy 2752 | Copied 2026-06-17 19:19:10

The booking was created fresh (not copied from another job — CopiedFromID is NULL) by David Livingston on 17 Jun 2026 at 19:19, the same day as the booking, MBL and HBL dates. The vessel name went into the HBL No field at initial data entry.

Root Cause
ff.BBooking_Import_HBL — HBLID 89323: HBLNo = 'MSC EMANUELA' (the vessel's name) Data entry error at booking creation. The vessel assigned to the booking (VesselID 5438) is named MSC EMANUELA, and that name was typed into the HBL No field on the Import Booking screen. ff.Update_BBooking_Import saves the screen payload verbatim with no validation on HBLNo, and the Invoice page displays the stored value as-is. Nothing is wrong with any procedure — the database value itself is wrong.
Solution

The correct HBL number exists nowhere in the system (CustomerRefNo and SAPNo are blank on the HBL record), so it must be supplied by the Dubai branch from the origin agent's documents. Once received, the fix is a one-row update. Invoices reference the HBL by HBLID, so no invoice rows need touching — the Invoice page reflects the correction immediately. 

-- PREVIEW (expect exactly 1 row: HBLID 89323, HBLNo 'MSC EMANUELA')
SELECT HBLID, BookingID, HBLNo, HBLDate
FROM ff.BBooking_Import_HBL
WHERE HBLID = 89323;

-- FIX (expect: 1 row affected; HBLNo is varchar(30))
UPDATE ff.BBooking_Import_HBL
SET HBLNo = '<CORRECT_HBL_NO>'
WHERE HBLID = 89323 AND HBLNo = 'MSC EMANUELA';

-- REVERT (if ever needed)
UPDATE ff.BBooking_Import_HBL
SET HBLNo = 'MSC EMANUELA'
WHERE HBLID = 89323;
Fix pending the correct HBL number from the Dubai branch Requested the real HBL No for job 2609977 / MBL MEDUYF745606 from the branch. The AND HBLNo = 'MSC EMANUELA' guard makes the update a no-op if the row has already been corrected in the meantime. Verify on the Invoice page after applying.
