Issue 1 — Vendor invoice does not appear on the Payment screen

Vendor: NEW GLOBE LOGISTIK LLP  |  Vendor Invoice: AHMAE/0050/26-27  |  Type: Data Change Request

Key Takeaway When a vendor invoice will not show on the Payment screen even though the vendor exists and is correctly linked, check the party on the job cost line, not the vendor master. The Payment screen matches invoices on BB_Cost.PartyID = FVendor.OpsID. A duplicate masterOps.Customer record (two ops IDs for the same firm, only one wired to a finance vendor) will strand every invoice that happens to be booked against the un-linked twin.
Problem

Vendor invoice AHMAE/0050/26-27 for vendor NEW GLOBE LOGISTIK LLP does not appear on the Transaction › Transaction › Payment screen, so it cannot be selected or paid. The task was raised as a Data Change Request with the subject note “VENDOR CODE NOT SHOW,” suggesting the vendor code / linkage as the suspect. Branch and BLISS-side job were not supplied and had to be traced from the vendor and vendor-invoice number.

Investigation
Step 1 — Identify the SP that powers the Payment screen's invoice list

The Payment screen first loads the vendor via Fin.Get_Vendor_Creditor (creditor ledgers under ParentID=52, returning each vendor's FVendor.OpsID), then loads that vendor's outstanding invoices via Fin.Search_Purchase_Invoice_Inventory_By_VendorID. The key mechanic in that SP is the vendor gate — the invoice is matched to the vendor through the cost line's PartyID, not the invoice header:

inner join @tblVendor tv  on tv.VendorID = bc.PartyID   -- @VendorID = FVendor.OpsID
left  join fin.FVendor v  on v.OpsID    = bc.PartyID

So an invoice appears only when a cost line exists with VendorInvNo = InvoiceNo, CostID = detail.CostID, and PartyID = the selected vendor's OpsID. Any mismatch on PartyID silently drops the row via the inner join.

Step 2 — Locate the vendor invoice header

Found the purchase invoice in ff.Tally_Invoice_Head_Pur:

SELECT TallyPurID, Branch, InvoiceNo, InvDate, JobID, TDSAmount
FROM ff.Tally_Invoice_Head_Pur
WHERE InvoiceNo LIKE '%AHMAE/0050/26-27%';
TallyPurID Branch InvoiceNo InvDate JobID TDSAmount 373774 Q AHMAE/0050/26-27 2026-06-16 82438 8738.00

The invoice exists, on branch Q and job 82438. So the invoice is booked — the problem is that the Payment screen is filtering it out, not that it is missing.

Step 3 — Verify the vendor master and its ops linkage

Because the subject pointed at “vendor code,” checked whether NEW GLOBE LOGISTIK LLP exists as a creditor ledger and whether it is linked to a finance vendor with a valid OpsID:

SELECT l.LedgerID, l.LedgerName, l.ParentID, l.isActive,
       v.CustomerID AS FVendor_CustomerID, v.OpsID, v.LedgerID, v.isActive
FROM fin.LedgerMaster l
LEFT JOIN fin.FVendor v ON v.LedgerID = l.LedgerID
WHERE l.LedgerName LIKE '%NEW GLOBE LOGISTIK%';
LedgerID LedgerName ParentID Active FVendor_Cust OpsID FV_Ledger FV_Active 12104 NEW GLOBE LOGISTIK LLP 52 1 40210 43472 12104 1

The vendor master is fully intact — active Sundry-Creditor ledger 12104, finance vendor 40210, linked to OpsID 43472. So this is not an unlinked / missing vendor code. The Payment screen would pass @VendorID = 43472 cleanly; the invoice must be dropped deeper inside the SP.

Step 4 — Get the invoice detail (segment) and rule out the branch gate

The invoice detail gives the segment (which decides the cost table) and the cost IDs:

SELECT TallyPurID, CostID, Segment, Amount
FROM ff.Tally_Invoice_Detail_Pur
WHERE TallyPurID = 373774;
CostID Segment Amount 0 E8 39312.72 0 E8 39312.72 375636 E8 225.00 375637 E8 454.00 375638 E8 685.00 375641 E8 435444.00

Segment E8 = Export, so the invoice routes through the ff.BB_Cost union of the SP, matching on cost IDs 375636 / 375637 / 375638 / 375641 (the two CostID 0 rows are GST/rounding lines and never match). The SP's branch gate joins only masterCom.OrganizationUnit rows with CompanyId=1 — confirmed branch Q = Ahmedabad, CompanyId=1, INR, so the branch gate is not the cause.

Step 5 — Inspect the actual cost lines — the mismatch appears

Looked directly at the export cost lines for this invoice:

SELECT CostID, JobID, VendorInvNo, PartyID
FROM ff.BB_Cost
WHERE JobID = 82438
  AND VendorInvNo LIKE '%AHMAE/0050/26-27%';
CostID JobID VendorInvNo PartyID 375636 82438 AHMAE/0050/26-27 44581 375637 82438 AHMAE/0050/26-27 44581 375638 82438 AHMAE/0050/26-27 44581 375641 82438 AHMAE/0050/26-27 44581

The cost lines match the invoice perfectly on VendorInvNo and CostID — but they carry PartyID = 44581, while the vendor is linked to OpsID = 43472. Because the SP does inner join @tblVendor on VendorID = bc.PartyID with @VendorID = 43472, 44581 ≠ 43472 drops every line.

Step 6 — Identify both ops parties — a duplicate customer

Checked what 43472 and 44581 are, and whether either has a finance vendor:

SELECT CustomerID, CuName, isActive FROM masterOps.Customer
WHERE CustomerID IN (43472, 44581);

SELECT CustomerID, CuName, OpsID, LedgerID, isActive FROM fin.FVendor
WHERE OpsID IN (43472, 44581);
masterOps.Customer 43472 NEW GLOBE LOGISTIK LLP 1 44581 NEW GLOBE LOGISTIK LLP 1 fin.FVendor 40210 NEW GLOBE LOGISTIK LLP OpsID 43472 Ledger 12104 Active 1

Both ops IDs are the same firm — a duplicate masterOps.Customer record. Only 43472 is wired to a finance vendor; 44581 has no FVendor row at all. The job costs were booked against the orphan twin 44581, which is payable to no one on the finance side.

Step 7 — Blast radius of the duplicate party

Checked how widely the orphan party 44581 is used:

SELECT JobID, VendorInvNo, COUNT(*) AS Lines
FROM ff.BB_Cost
WHERE PartyID = 44581
GROUP BY JobID, VendorInvNo
ORDER BY JobID;
JobID VendorInvNo Lines 62389 AHMAE/0045/25-26 1 72032 CCUAE/0096/25-26 1 72076 BOMAE/1781/25-26 1 74390 BOMAE/2603/25-26 1 74783 CCUAE/0138/25-26 1 80736 AHMAE/0024/26-27 5 82438 AHMAE/0050/26-27 4 ← reported

The orphan party stranded 7 vendor invoices across 7 jobs, all NEW GLOBE LOGISTIK LLP. The reported invoice is one of them; the other six have the same defect but were not raised, so only the reported invoice was corrected (see Recurrence Risk).

Root Cause
ff.BB_Cost (job 82438, invoice AHMAE/0050/26-27) — PartyID = 44581, should be 43472 masterOps.Customer holds two records for NEW GLOBE LOGISTIK LLP — 43472 and 44581. The finance vendor (FVendor 40210 → ledger 12104) is linked to OpsID 43472, but this job's cost lines were booked against the duplicate 44581, which has no finance vendor. The Payment screen SP Fin.Search_Purchase_Invoice_Inventory_By_VendorID gates invoices on inner join @tblVendor on VendorID = bc.PartyID (@VendorID = OpsID = 43472), so all four cost lines — and therefore the whole invoice — are dropped.
Confirmation Queries

Preview of the exact rows to change (returned 4 rows, all PartyID = 44581):

SELECT CostID, JobID, VendorInvNo, PartyID
FROM ff.BB_Cost
WHERE CostID IN (375636, 375637, 375638, 375641);
375636 82438 AHMAE/0050/26-27 44581 375637 82438 AHMAE/0050/26-27 44581 375638 82438 AHMAE/0050/26-27 44581 375641 82438 AHMAE/0050/26-27 44581
Solution

Repointed the four export cost lines from the orphan duplicate party 44581 to the correctly-linked party 43472. Same firm, same amounts and currency — only the party pointer changes to the ops record that owns the Sundry-Creditor ledger. Scoped to the four exact CostIDs so it is surgical and cleanly reversible. Applied and verified in the frontend on 22 Jul 2026 — the invoice now appears and is selectable on the Payment screen under NEW GLOBE LOGISTIK LLP. 

-- Applied (4 rows affected)
UPDATE ff.BB_Cost
SET PartyID = 43472
WHERE CostID IN (375636, 375637, 375638, 375641)
  AND PartyID = 44581;
Revert (if needed) 
UPDATE ff.BB_Cost
SET PartyID = 44581
WHERE CostID IN (375636, 375637, 375638, 375641)
  AND PartyID = 43472;
Permanent fix (pending — master-data cleanup) The durable fix is at the master level: repoint the remaining six invoices' BB_Cost lines from 44581 to 43472 (with Finance sign-off, since some are FY 25-26 and may already be reconciled), then deactivate / merge duplicate ops-customer 44581 so new jobs can no longer be booked against it. Only the reported invoice (job 82438) was corrected under this task.

The two CostID = 0 rows in the invoice detail (Step 4) are GST / rounding lines, not job cost lines. They never carry a BB_Cost.CostID and are expected to be absent from the party match — they are not part of the defect.
