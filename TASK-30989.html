Issue 1 — Invoice Numbers Not Appearing on Job Cost Sheet

Job: 2603498  |  Branch: Antwerp (W)  |  BookingID: 81203

Key Takeaway When a booking has multiple HBLs, the invoice can be finalized with the wrong Invoice_Rate.JobId. The SP joins ir.JobId = r.HBLID — if these don't match, the invoice number is silently absent from the cost sheet. Fixing JobId alone is not enough: when revenue is first entered, unlinked Invoice_Rate records (InvoiceID = NULL) are created at the correct HBLID. If the system later creates new Invoice_Rate records at the wrong HBLID instead of updating the originals, both sets end up active after the fix — causing each revenue row to appear twice (once with an invoice, once without). The orphaned unlinked records must also be deactivated.
Problem

Invoice numbers were not appearing in the revenue rows of the Job Cost Sheet for Job 2603498 (Antwerp branch). At the time of investigation, it was not yet known which invoices were affected or whether invoices had been raised at all — the only symptom was blank Vendor Inv cells.

The cost sheet is generated by calling ff.Costsheet_I @ID=81203 via GET /api/master-api/report/getCostSheetHtml?JobID=81203&spname=ff.Costsheet_I. The API response showed empty Vendor Inv cells for all revenue charges — with one exception: Emergency Bunker Surcharge was displaying correctly with an invoice number. This told us the issue was selective, not a blanket failure, and that the SP logic itself was working for at least some records.

Investigation
Step 1 — Get BookingID and HBLIDs for the job

The SP uses @ID as both @BookingID and populates a table variable with the HBLIDs. First confirmed the booking structure.

SELECT b.BookingID, b.BookingNo, h.HBLID, h.HBLNo
FROM ff.BBooking_Import b
JOIN ff.BBooking_Import_HBL h ON h.BookingID = b.BookingID
WHERE b.BookingNo = '2603498' AND b.isActive = 1
BookingID BookingNo HBLID HBLNo 81203 2603498 83970 2603498 81203 2603498 84462 7260094

This booking has two HBLs: HBLID 83970 (primary HBL 2603498) and HBLID 84462 (HBL 7260094).

Step 2 — Discover which invoices exist for this job and check their status

We did not yet know which invoices were linked to this job. The goal of this step was to find them and verify whether their status flags could be causing the SP to exclude them.

The initial approach was to query ff.Invoice_Rate directly with a LEFT JOIN to ff.Invoice filtering on both HBLIDs — but this returned 300,000+ rows due to the size of ff.Invoice_Rate. We reversed the direction, starting from ff.Invoice and joining to ff.Invoice_Rate, which kept the result set manageable. 

SELECT
    i.InvoiceID, i.InvoiceNumber, i.InvoiceDate,
    i.InvStatus            AS Invoice_InvStatus,
    ir.InvRateId, ir.JobId, ir.RevenueID,
    ir.Segment, ir.isActive AS IR_isActive,
    ir.InvStatus           AS IR_InvStatus
FROM ff.Invoice i
JOIN ff.Invoice_Rate ir ON ir.InvoiceID = i.InvoiceID
WHERE ir.JobId IN (83970, 84462)
  AND LEFT(ir.Segment, 1) = 'I'
ORDER BY i.InvoiceDate

This is where the specific invoice numbers were identified for the first time: 26704183 (InvoiceID 239616) and 26704184 (InvoiceID 239617), both dated 27 May 2026. All four invoices have InvStatus = 11 and IR_isActive = 1 — both pass the SP filters, so the status flags were not the cause.

The key observation was the JobId column: the EBS invoices (26704927/26704928) that were displaying correctly had JobId = 84462, while the missing invoices (26704183/26704184) had JobId = 83970. Since the SP joins ir.JobId = r.HBLID, this difference in JobId was the immediate suspect.

Step 3 — Check where IRevenue records actually live

The SP joins ir.JobId = r.HBLID to match Invoice_Rate to IRevenue. Checked which HBLID the revenue records are stored under. 

-- IRevenue counts by HBLID
SELECT r.HBLID, COUNT(*) AS RevenueCount
FROM ff.IRevenue r
WHERE r.HBLID IN (83970, 84462) AND r.isActive = 1
GROUP BY r.HBLID

-- Invoice_Rate counts by JobId
SELECT ir.JobId, COUNT(*) AS InvRateCount
FROM ff.Invoice_Rate ir
WHERE ir.JobId IN (83970, 84462)
  AND LEFT(ir.Segment, 1) = 'I' AND ir.isActive = 1
GROUP BY ir.JobId
-- IRevenue HBLID RevenueCount 84462 34 -- Invoice_Rate JobId InvRateCount 83970 32 84462 34

All 34 IRevenue records live under HBLID 84462 — zero under 83970. But the 32 Invoice_Rate records for the main invoices use JobId = 83970. The SP join ir.JobId = r.HBLID compares 83970 ≠ 84462 → returns NULL → invoice number never appears.

Step 4 — Confirm the RevenueIDs would match after correcting JobId

Before applying the fix, verified that the RevenueID values in Invoice_Rate (pointing to 83970) actually match IRevenueID values in IRevenue under 84462.

SELECT COUNT(*) AS MatchCount
FROM ff.Invoice_Rate ir
JOIN ff.IRevenue r ON r.IRevenueID = ir.RevenueID
                   AND r.HBLID = 84462
                   AND r.isActive = 1
WHERE ir.JobId = 83970
  AND LEFT(ir.Segment, 1) = 'I'
  AND ir.isActive = 1
MatchCount 32

All 32 records match. Updating JobId from 83970 to 84462 is sufficient — no other fields need changing.

Root Cause
ff.Invoice_Rate — InvoiceIDs 239616, 239617: JobId = 83970 (should be 84462) The invoices 26704183 and 26704184 were finalized with Invoice_Rate.JobId = 83970 (the primary HBL), but all ff.IRevenue records for this job are stored under HBLID 84462 (the second HBL, 7260094). The SP ff.Costsheet_I joins ir.JobId = r.HBLID — since 83970 ≠ 84462, the join returns NULL and the invoice numbers are silently excluded from the cost sheet output.
Solution
Fix 1 — Correct JobId on the mislinked Invoice_Rate records (01 Jul 2026)

Preview the 32 rows to be updated:

SELECT ir.InvRateId, ir.JobId, ir.RevenueID, ir.InvoiceID, i.InvoiceNumber
FROM ff.Invoice_Rate ir
JOIN ff.Invoice i ON i.InvoiceID = ir.InvoiceID
WHERE ir.JobId = 83970
  AND LEFT(ir.Segment, 1) = 'I'
  AND ir.isActive = 1
  AND ir.InvoiceID IN (239616, 239617)
UPDATE ff.Invoice_Rate
SET JobId = 84462
WHERE JobId = 83970
  AND LEFT(Segment, 1) = 'I'
  AND isActive = 1
  AND InvoiceID IN (239616, 239617)
(32 rows affected)

After this fix, invoice numbers appeared in the cost sheet — but revenue rows were now duplicated (66 rows instead of 34, total doubled from ~5986 to ~11689 EUR). Each charge appeared twice: once with an invoice number, once with an empty cell.

Step 5 — Diagnose row duplication after Fix 1

The original 34 Invoice_Rate records for JobId = 84462 included 32 unlinked records (InvoiceID = NULL) created when revenue was first entered. Instead of updating these when the invoices were raised, the system created 32 new records at JobId = 83970. After Fix 1, both sets were active under JobId = 84462, so the SP join returned two rows per revenue line.

Verified the count of orphaned unlinked duplicates:

SELECT COUNT(*) AS UnlinkedDuplicates
FROM ff.Invoice_Rate
WHERE JobId = 84462
  AND LEFT(Segment, 1) = 'I'
  AND isActive = 1
  AND InvoiceID IS NULL
  AND RevenueID IN (
      SELECT RevenueID FROM ff.Invoice_Rate
      WHERE JobId = 84462
        AND LEFT(Segment, 1) = 'I'
        AND isActive = 1
        AND InvoiceID IN (239616, 239617)
  )
UnlinkedDuplicates 32

Exactly 32 orphaned records — one per revenue line, all with InvoiceID = NULL. These are the original unlinked rate records that should have been superseded when the invoice was created.

Fix 2 — Deactivate the orphaned unlinked records (01 Jul 2026)
UPDATE ff.Invoice_Rate
SET isActive = 0
WHERE JobId = 84462
  AND LEFT(Segment, 1) = 'I'
  AND isActive = 1
  AND InvoiceID IS NULL
  AND RevenueID IN (
      SELECT RevenueID FROM ff.Invoice_Rate
      WHERE JobId = 84462
        AND LEFT(Segment, 1) = 'I'
        AND isActive = 1
        AND InvoiceID IN (239616, 239617)
  )
(32 rows affected)

After Fix 2, all 34 revenue rows appear exactly once on the cost sheet, each with its correct invoice number. Revenue total returned to the expected value. The InvoiceID IS NULL filter ensures EBS records (InvoiceIDs 243117/243118) are not touched.

To revert both fixes — order is critical Two queries are required and must be run in this exact order. Step 1 must run before Step 2 because Step 1's subquery identifies the orphaned records by matching their RevenueID against the linked records (InvoiceID IN (239616, 239617)) that are still at JobId = 84462. If Step 2 runs first, those linked records move back to JobId = 83970 and Step 1's subquery returns nothing — the orphaned records remain deactivated and the revert is incomplete. 
-- Step 1 FIRST: re-activate the 32 deactivated unlinked records
-- (must run before Step 2 — subquery relies on linked records still being at JobId=84462)
UPDATE ff.Invoice_Rate
SET isActive = 1
WHERE JobId = 84462
  AND LEFT(Segment, 1) = 'I'
  AND isActive = 0
  AND InvoiceID IS NULL
  AND RevenueID IN (
      SELECT RevenueID FROM ff.Invoice_Rate
      WHERE JobId = 84462
        AND LEFT(Segment, 1) = 'I'
        AND isActive = 1
        AND InvoiceID IN (239616, 239617)
  )
-- Expected: 32 rows affected

-- Step 2 AFTER: revert JobId back to 83970
UPDATE ff.Invoice_Rate
SET JobId = 83970
WHERE JobId = 84462
  AND LEFT(Segment, 1) = 'I'
  AND isActive = 1
  AND InvoiceID IN (239616, 239617)
-- Expected: 32 rows affected
Observations

After both fixes, the cost sheet for job 2603498 shows two cost rows with empty Vendor Inv cells. These are unrelated to this investigation — they are cost entries where the vendor invoice has not yet been received and entered in the system. The cost sheet correctly shows a blank Vendor Inv for any charge not yet linked to a vendor invoice.
