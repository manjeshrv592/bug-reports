1 — The data model in five tables

An import shipment in BLISS involves these tables (all in schema ff unless noted):

TableOne row per…Key columns for this case
BBooking_ImportImport booking (the job)BookingID, BookingNo
BBooking_Import_HBLHouse Bill of Lading under a booking — one booking can have several HBLsHBLID, BookingID, HBLNo
IRevenueRevenue (charge) line, attached to one specific HBLIRevenueID, HBLID, ChargeID
InvoiceCustomer invoice headerInvoiceID, InvoiceNumber, JobId
Invoice_RateInvoice line — the bridge between a revenue line and an invoiceInvRateId, InvoiceID, RevenueID, JobId, Segment

The linkage that matters here:

BBooking_Import (81203, job 2603498) ├─ BBooking_Import_HBL HBLID 83970 (HBL 2603498 — "primary") └─ BBooking_Import_HBL HBLID 84462 (HBL 7260094) └─ IRevenue ×34 (ALL revenue lines live here: HBLID = 84462) ▲ │ Invoice_Rate.RevenueID ──► IRevenue.IRevenueID │ Invoice_Rate.JobId ──► must equal that IRevenue row's HBLID │ Invoice_Rate.InvoiceID ──► Invoice.InvoiceID Invoice 26704183 / 26704184 (the two invoices whose numbers went missing)

A revenue line becomes "invoiced" when an Invoice_Rate row exists that points at it (RevenueID) and at an invoice (InvoiceID). When revenue is first entered, the system creates Invoice_Rate rows with InvoiceID = NULL ("unlinked"); finalizing an invoice is supposed to fill in the InvoiceID on those existing rows (section 5 shows the exact code that does this).

2 — What "JobId" really means: a column with six meanings

The name Invoice_Rate.JobId suggests it identifies a job. It does not. Its meaning depends on the Segment of the row. This was established by scanning every join involving this column across the full SP export (138 of 2,274 BSS objects reference Invoice_Rate). The census of what Invoice_Rate.JobId is joined to:

SegmentJobId is joined to…Times seenExample SPs (from source export)
Export (E%)ff.BBL.BLID — a Bill of Lading ID28BInvoice_search, Costsheet_E
Import (I%)ff.BBooking_Import_HBL.HBLID and ff.IRevenue.HBLID — an HBL ID11 + 10Costsheet_I, Dashboard_Import, Engine_Jobwise_PnL_Live, Get_Invoice_Correction, Fn_FrightRate, Get_Jobwise_Cost_Revenue_Listing

Consequence For an import invoice line, "changing JobId" does not change which job the line belongs to. Both candidate values (83970 and 84462) are HBLs of the same booking 81203. The question is only: which HBL of that booking should the line point at — and the code in sections 3 and 5 answers that unambiguously.
3 — How the cost sheet is generated, and where it breaks

The frontend calls GET /api/master-api/report/getCostSheetHtml?JobID=81203&spname=ff.Costsheet_I, which executes ff.Costsheet_I @ID = 81203. Inside the SP, @ID is the BookingID, and the revenue section iterates over all HBLs of the booking:

ff.Costsheet_I — verbatim from source export (top of procedure)
declare @BookingID  int=@ID, @JobID int=@ID
declare @tabBook table (BookingID int)
insert into @tabBook select HBLID from ff.BBooking_Import_HBL where BookingID=@BookingID

So @tabBook = {83970, 84462}. The revenue rows are then selected and the invoice number attached:

ff.Costsheet_I — verbatim, revenue block
from ff.IRevenue r
...
inner join @tabBook tb          on tb.BookingID=r.HBLID
...
left join ff.Invoice_Rate ir    on ir.JobId=r.HBLID and ir.RevenueID=r.IRevenueID
                                and left(ir.Segment,1)='I' and ir.isActive = 1
left join ff.Invoice i          on i.InvoiceID=ir.InvoiceID

Read that join carefully — it is the heart of the whole case:

  • Revenue rows are found via r.HBLID — all 34 live at HBLID 84462, so all 34 appear on the sheet. The revenue lines themselves were never missing.
  • The invoice number comes through a LEFT JOIN that requires both ir.JobId = r.HBLID and ir.RevenueID = r.IRevenueID.
  • Our 32 broken rows had the right RevenueID but JobId = 83970 ≠ 84462. First condition fails → LEFT JOIN yields NULL → i.InvoiceNumber is NULL → the Vendor Inv / Invoice cell renders blank. No error, no missing row — just a silently empty cell.
Why one charge (EBS) still showed its invoice number The Emergency Bunker Surcharge invoices (26704927/26704928) were raised through the correct path and their Invoice_Rate rows carry JobId = 84462. Both join conditions pass for them. This working "control group" on the very same job is what proved the SP logic itself was fine and the data was at fault.
4 — The evidence trail: four queries that isolate the fault

All queries were run on production during the investigation (01 Jul 2026); results shown are the actual outputs.

Evidence 1 — The booking has two HBLs
SELECT b.BookingID, b.BookingNo, h.HBLID, h.HBLNo
FROM ff.BBooking_Import b
JOIN ff.BBooking_Import_HBL h ON h.BookingID = b.BookingID
WHERE b.BookingNo = '2603498' AND b.isActive = 1
BookingID BookingNo HBLID HBLNo 81203 2603498 83970 2603498 81203 2603498 84462 7260094

Established: this is a multi-HBL booking. Two candidate HBLIDs exist — the precondition for the whole bug.

Evidence 2 — Four invoices exist; the visible pair and the missing pair differ in exactly one column
SELECT i.InvoiceID, i.InvoiceNumber, i.InvoiceDate, i.InvStatus,
       ir.InvRateId, ir.JobId, ir.RevenueID, ir.Segment, ir.isActive, ir.InvStatus
FROM ff.Invoice i
JOIN ff.Invoice_Rate ir ON ir.InvoiceID = i.InvoiceID
WHERE ir.JobId IN (83970, 84462) AND LEFT(ir.Segment, 1) = 'I'
ORDER BY i.InvoiceDate
InvoiceID InvoiceNumber InvoiceDate InvStatus JobId Segment isActive rows 239616 26704183 2026-05-27 11 83970 I1 1 x16 ← MISSING on sheet 239617 26704184 2026-05-27 11 83970 I1 1 x16 ← MISSING on sheet 243117 26704927 2026-06-16 11 84462 I1 1 x1 ← visible (EBS) 243118 26704928 2026-06-16 11 84462 I1 1 x1 ← visible (EBS)

Established: every status flag is identical between the visible and missing invoices (InvStatus = 11, isActive = 1, same segment). The only difference is JobId: 84462 on the pair that displays, 83970 on the pair that does not. Given the join in section 3, this is the prime suspect — and it also rules out every "status/filter" theory in one query.

Evidence 3 — All revenue lives at 84462; nothing lives at 83970
SELECT r.HBLID, COUNT(*) FROM ff.IRevenue r
WHERE r.HBLID IN (83970, 84462) AND r.isActive = 1 GROUP BY r.HBLID

SELECT ir.JobId, COUNT(*) FROM ff.Invoice_Rate ir
WHERE ir.JobId IN (83970, 84462) AND LEFT(ir.Segment,1)='I' AND ir.isActive = 1 GROUP BY ir.JobId
IRevenue: HBLID 84462 → 34 rows (HBLID 83970 → ZERO rows) Invoice_Rate: JobId 83970 → 32 rows + JobId 84462 → 34 rows

Established: IRevenue has zero rows at 83970. Therefore the 32 Invoice_Rate rows sitting at JobId = 83970 can never satisfy ir.JobId = r.HBLID for any revenue row of this booking. They are unreachable by design.

Evidence 4 — The broken rows internally contradict themselves

This is the decisive query. It asks: do the RevenueIDs of the rows at 83970 point at revenue that lives at 84462?

SELECT COUNT(*) FROM ff.Invoice_Rate ir
JOIN ff.IRevenue r ON r.IRevenueID = ir.RevenueID
                   AND r.HBLID = 84462 AND r.isActive = 1
WHERE ir.JobId = 83970 AND LEFT(ir.Segment,1) = 'I' AND ir.isActive = 1
MatchCount: 32 (all of them)

Established: every one of the 32 rows says two incompatible things at once — its RevenueID points at revenue on HBL 84462, while its JobId claims HBL 83970. One of the two fields must be wrong. RevenueID is the hard link (it decides which charge the money belongs to, and matches the amounts invoiced); JobId is the derived locator. The locator was wrong.

5 — Why the corrupt rows existed: ff.Update_Invoice walkthrough

Invoice finalization is done by ff.Update_Invoice. The screen posts the selected charge lines as a table variable @Invoice_Rate (each row carrying a JobID from the page context). Two statements matter, quoted verbatim from the source export:

ff.Update_Invoice — verbatim: inserts payload rows that have no existing InvRateId
insert into ff.Invoice_Rate (InvoiceID, ..., JobID, Segment, ..., RevenueID, InvStatus)
select null, ..., r.JobID, r.Segment, ..., r.RevenueID, 11
from @Invoice_Rate r
left join ff.Invoice_Rate a on a.InvRateId=r.InvRateId and a.ContainerType=r.ContainerType
where isNull(r.InvRateId,0)=0
ff.Update_Invoice — verbatim: links unlinked rows to the newly created invoice
Update ff.Invoice_Rate set InvoiceId=@InvoiceIdentity, ...
from ff.Invoice_Rate u, @Invoice_Rate r
where r.JobId=u.JobID and r.isSelect=1 and isNull(u.InvoiceID,0)=0 and u.RevenueID=r.RevenueID

Now replay what happened when invoices 26704183/26704184 were finalized with a payload carrying JobID = 83970 (the primary HBL — the context the screen was opened under):

#What the SP didResult
1Payload rows had no InvRateId → the INSERT created 32 brand-new rows copying the payload's JobID = 83970, InvoiceID = NULL.32 new rows at 83970
2The UPDATE tried to link unlinked rows matching r.JobId = u.JobID AND u.RevenueID = r.RevenueID. The original unlinked rows (created with the revenue) sit at JobID = 84462 → 83970 ≠ 84462 → they were never matched.Original 32 rows orphaned (InvoiceID stays NULL)
3The freshly inserted rows (step 1) do match the payload on both fields → they got the new InvoiceID.32 linked rows at the wrong HBL

Predicted end state: 32 linked rows at 83970 + 32 orphaned unlinked rows at 84462 + the 2 correctly-made EBS rows at 84462. That is exactly the state found in Evidence 3 (32 + 34) and in the original report's Step 5 (32 orphans with InvoiceID IS NULL). The mechanism is fully explained by the SP's own code — note that the SP's matching logic itself assumes the invariant JobId ↔ RevenueID's HBLID; when the payload violates it, the SP silently duplicates instead of linking.

The real bug is upstream ff.Update_Invoice trusts the JobID the screen sends. When the invoicing screen is used on a multi-HBL booking under the primary HBL's context while the revenue lives on another HBL, the payload carries the wrong HBLID. That is the recurring defect; the database rows are only its symptom.
6 — The fix, and the proof that changing JobId was correct

The fix (01 Jul 2026, full queries in TASK-30989.html) had two parts:

  1. Fix 1: UPDATE ff.Invoice_Rate SET JobId = 84462 on the 32 linked rows (InvoiceIDs 239616/239617) — restoring the invariant.
  2. Fix 2: deactivate (isActive = 0) the 32 orphaned duplicates (InvoiceID IS NULL) that step 2 of section 5 left behind — removing the double-count.

Proof of correctness, each item independently verifiable:

ProofStatementSource
P1 For import lines, JobId is an HBL locator, not a job identifier — both values belong to the same booking, so no cross-job data movement occurred. Join census across all 4,612 SPs/views (section 2 table).
P2 The system's own code defines the correct value: Costsheet_I joins ir.JobId = r.HBLID AND ir.RevenueID = r.IRevenueID, and Update_Invoice links on r.JobId = u.JobID AND u.RevenueID = r.RevenueID. Both require JobId = the HBLID of the revenue row referenced by RevenueID. Our rows referenced revenue at 84462 → the only self-consistent value is 84462. Verbatim SP source, sections 3 and 5.
P3 Control group: the EBS invoices on the same job, created through the correct flow, already carry JobId = 84462. The fix made the broken rows identical in form to rows the system itself produces when healthy. Evidence 2 result set.
P4 Post-fix behavior matched prediction exactly: after Fix 1 every revenue line showed its invoice number but appeared twice (34 → 66 rows — predicted, because the orphans now also matched); after Fix 2 the sheet returned to 34 rows, each with the correct invoice number and the original total. Frontend verification, 01 Jul 2026 (documented in TASK-30989.html).
7 — What could the fix have broken? Blast-radius audit

The strongest version of the team's concern is: "other code might rely on JobId being 83970." This was checked against every SP and view in the database export:

ConcernCheckedFinding
Does anything require the invoice header's Invoice.JobId to equal the line's Invoice_Rate.JobId? All join conditions between the two tables, all objects No. Zero occurrences of i.JobId = ir.JobId anywhere. Every SP connects them via ir.InvoiceId = i.InvoiceID only. The header (which drives invoice search and listings) was not modified and is unaffected.
Does invoice printing read Invoice_Rate.JobId? ff.Rpt_Invoice source — every reference to Invoice_Rate in it No. All 10+ joins in the print SP are ir.InvoiceId = i.InvoiceID. Printed invoices are byte-identical before/after the fix.
Which SPs do read ir.JobId for import rows, and what happens to them? Full join census (section 2) Costsheet_I, Dashboard_Import, Get_Jobwise_Cost_Revenue_Listing, Engine_Jobwise_PnL_Live, Get_Invoice_Correction, Fn_FrightRate. All of them join it to an HBLID of revenue/HBL rows — i.e. all of them were seeing NULLs or misattribution while the value was 83970, and all now resolve to the HBL where the revenue actually lives. For these SPs the fix is the correction, not a side effect. Job/booking-level totals are unchanged either way since both HBLs belong to booking 81203.
Was any money value, charge, quantity, or tax figure touched? The two UPDATE statements No. Fix 1 changed one locator column on 32 rows; Fix 2 flipped isActive on 32 rows that were never linked to any invoice. Amount columns untouched; both are reversible (revert scripts in TASK-30989.html, order-sensitive).
8 — Alternatives considered and rejected
AlternativeWhy it was rejected
Do nothing ("don't touch data") The 32 rows violate the invariant the system's own code enforces. The cost sheet (and every SP in section 7's third row) stays permanently wrong for this job. There is no code change that can make a self-contradictory row correct.
Move the revenue instead: UPDATE ff.IRevenue SET HBLID = 83970 Would relocate all 34 revenue lines to the primary HBL — falsifying where the revenue belongs, and instantly breaking the two EBS invoices (243117/243118) whose correctly-created rows point at 84462. Fixes two invoices by corrupting two others.
Change the SP join (make Costsheet_I ignore JobId, join on RevenueID alone) A code change to accommodate corrupt data. RevenueID alone is not guaranteed unique across segments/history in this schema (the join deliberately uses both fields), and every other SP in the census would still see the wrong HBL. Also requires dev/deployment cycle for what is a 32-row data defect.
Relink the orphans instead (set their InvoiceID, deactivate the 83970 rows) Legitimate equivalent — the end state is the same invariant-consistent shape. It touches the same number of rows (64) and has no advantage; the chosen fix has the smaller diff on the rows that carry the invoice link.
9 — Detection query: find every other affected job

Because the upstream defect (section 5) still exists, other multi-HBL import bookings may already carry the same corruption. This query finds every active import invoice line whose JobId disagrees with the HBLID of the revenue row it references — i.e. every row in the same broken state this task fixed: 

SELECT ir.InvRateId, ir.InvoiceID, i.InvoiceNumber,
       ir.JobId        AS RateJobId,
       r.HBLID         AS RevenueHBLID,
       h.BookingID, b.BookingNo
FROM ff.Invoice_Rate ir
JOIN ff.IRevenue r             ON r.IRevenueID = ir.RevenueID AND r.isActive = 1
LEFT JOIN ff.Invoice i         ON i.InvoiceID = ir.InvoiceID
LEFT JOIN ff.BBooking_Import_HBL h ON h.HBLID = r.HBLID
LEFT JOIN ff.BBooking_Import b     ON b.BookingID = h.BookingID
WHERE LEFT(ir.Segment, 1) = 'I'
  AND ir.isActive = 1
  AND ir.JobId <> r.HBLID
ORDER BY b.BookingNo, ir.InvoiceID;
How to use Run periodically (or after complaints of blank invoice numbers on import cost sheets). Each returned row is a candidate for the same two-part fix as TASK-30989 — but verify per job, following the evidence steps in section 4, before updating anything. Rows with InvoiceID IS NULL in the result are potential orphans; rows with an InvoiceNumber are the mislinked lines. If this query starts returning new rows regularly, that is the signal to prioritize the frontend/payload fix in the invoicing screen.
