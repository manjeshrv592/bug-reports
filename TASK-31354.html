Issue 1 — Invoice Number Generated as “to insert”

Job No: 2611933  |  Branch: Bangkok (B)  |  Module: Invoice › Invoice › Invoice

Key Takeaway When an invoice number, purchase entry number, or job number saves as a literal word (to insert, toinsert, or ParameterErr), it is not a UI bug — it is the appConfig.Get_Number sentinel being persisted verbatim. to insert specifically means the branch’s number series exists but no row covers the invoice date, i.e. the annual financial-year window was never rolled over. Every branch series carries a DateTo; at each year boundary the next window must exist or the next invoice strands.
Problem

On Invoice › Invoice › Invoice (staging, Bangkok branch), the invoice for job 2611933 displayed the invoice number as the literal string “to insert” instead of a real number, and the format did not match the other invoices in the list. All other rows on the same screen showed proper numbers. A permanent fix was requested, plus a real invoice number for this invoice.

Investigation
Step 1 — Trace the number-generation path in the SP

The invoice grid / generation is served by ff.Update_Invoice. It builds the number with NType = 'FINV', a numbering segment of X (taxable), N (GST = 0) or Z (SEZ), and a branch from ff.Fn_BillBranch — then writes whatever comes back straight into ff.Invoice.InvoiceNumber with no validation. 

-- ff.Update_Invoice (number block)
set @Segment = 'X'                         -- taxable
if @GSTAmt = 0  set @Segment = 'N'         -- zero GST
if @isSEZ   = 1 set @Segment = 'Z'         -- SEZ
exec appConfig.Get_Number @Branch, @Segment, 'FINV', @InvDate, '', @N output
insert into ff.Invoice (... InvoiceNumber ...) select ..., @N, ...   -- @N stored verbatim

-- appConfig.Get_Number (the sentinel)
if exists (... BranchId=@Branch and NType=@NType and Segment=@Segment and isActive=1)
begin
    if exists (... and @Date between DateFrom and DateTo ...)
        Update appConfig.Numbers set Number = Number + 1 where ...
    else
    begin
        set @R = 'to insert'   -- series exists, but NO row covers @Date
        set @Number = @R
        return                 -- returns BEFORE the trailing space-strip
    End
    ...
end
else
    set @R = 'Parameter Err'   -- no series row at all -> later stripped to 'ParameterErr'

The early return is why the value keeps its space (to insert) — unlike Parameter Err, which flows on to replace(@R,' ','') and becomes ParameterErr. So to insert = “an active series exists for this branch, but no year-window row covers the invoice date.”

Step 2 — Locate the invoice and confirm the DB / branch / segment

The staging header reads QA_BSS, but that is the app label — no such database exists. A probe across the databases found the row in BSS. 

SELECT 'BSS' AS Db, i.InvoiceId, i.InvoiceNumber, i.InvoiceDate, i.Segment, i.JobId
FROM BSS.ff.Invoice i
LEFT JOIN BSS.ff.miscJob m ON m.JobID = i.JobId
WHERE i.InvoiceNumber = 'to insert' AND m.JobNo = '2611933';
BSS 248205 to insert 2026-07-20 M3 2884

Invoice 248205, stored segment M3 (a Misc job, JobId 2884), invoice date 20-Jul-2026. The stored branch was B and the invoice’s revenue carried zero GST — so the numbering segment used was N. Note the M3 rule in the SP (if @Segment='M3' set @Branch=@pBranch): for M3 the numbering branch is the login branch, which here matched the job’s own branch B (Bangkok).

Step 3 — Inspect the Bangkok FINV series

List every B / FINV series row and check which window covers the invoice date.

SELECT BranchId, NType, Segment, YearCode, DateFrom, DateTo, Number, NFormat, isActive
FROM appConfig.Numbers
WHERE NType = 'FINV' AND BranchId = 'B'
ORDER BY Segment, DateFrom;
B FINV N 26 2025-04-01 2026-03-31 2 Branch Taxtype YYMM99999 1 B FINV X 26 2025-04-01 2026-03-31 0 Branch Taxtype YYMM99999 1 B FINV Z 26 2025-04-01 2026-03-31 0 Branch Taxtype YYMM99999 1 ... (rows for YearCode 20, 22, 23, 24, 25 above) ... NO row with DateFrom = 2026-04-01 / DateTo = 2027-03-31 for any segment

Bangkok uses a fiscal year (Apr–Mar). Every segment (N, X, Z) stops at 2026-03-31. The invoice dated 2026-07-20 falls in the next window (2026-04-01 → 2027-03-31), which was never created. Active rows exist for the series → the SP takes the to insert branch.

Step 4 — Rule out branch mis-stamping (why only this row failed)

The screen’s other invoices are SGN… (Singapore, code S → SG). To be sure 248205 was not simply stamped with the wrong branch, we compared how every branch’s M3 invoice is numbered. 

SELECT TOP 30 i.InvoiceNumber, i.Segment, i.InvoiceDate, i.Branch AS InvBranch,
       m.Branch AS JobBranch, m.BilledBy
FROM ff.Invoice i LEFT JOIN ff.miscJob m ON m.JobID = i.JobId
WHERE i.Segment = 'M3'
ORDER BY i.InvoiceDate DESC;
M (Mumbai) MX270701152 InvBranch=M JobBranch=M T (Tuticorin) TX270701516 InvBranch=T JobBranch=T DXB XN260700558 InvBranch=DXB JobBranch=DXB C CN270700184 InvBranch=C JobBranch=C W (Antwerp) 26705994 InvBranch=W JobBranch=W B (Bangkok) to insert InvBranch=B JobBranch=B <-- only failure

Every M3 invoice numbers under its own branch and InvBranch = JobBranch for all of them. Bangkok is the only branch that failed — and its job branch is genuinely B. So 248205 is correctly a Bangkok invoice; the “wrong format” impression came from comparing it against the unrelated Singapore (SGN) rows that share the 517-row list. The single fault is the missing Bangkok FY window.

Root Cause
appConfig.Numbers — no FY2026-27 window for BranchId 'B', NType 'FINV' Bangkok’s FINV number series was never rolled into the new fiscal year. The latest rows for all three segments (N, X, Z) end at 2026-03-31; no row exists for 2026-04-01 → 2027-03-31 (YearCode 27). Because an active series still exists for the branch, appConfig.Get_Number returns the sentinel to insert for any invoice dated after 31-Mar-2026, and ff.Update_Invoice saves that string as the invoice number with no validation.
Confirmation Queries

The queries that proved root cause, in order:

-- (a) the invoice lives in BSS, is Misc/M3, zero-GST -> numbering segment N, branch B
SELECT i.InvoiceId, i.InvoiceNumber, i.Segment, i.Branch, i.JobId
FROM ff.Invoice i LEFT JOIN ff.miscJob m ON m.JobID = i.JobId
WHERE i.InvoiceNumber = 'to insert' AND m.JobNo = '2611933';        -- 248205 / M3 / B

-- (b) B/FINV series has NO row covering the 20-Jul-2026 invoice date
SELECT BranchId, Segment, YearCode, DateFrom, DateTo, Number, NFormat
FROM appConfig.Numbers
WHERE NType = 'FINV' AND BranchId = 'B'
ORDER BY Segment, DateFrom;                                          -- newest window ends 2026-03-31

-- (c) every other branch's M3 invoice numbers under its own branch; only Bangkok stranded
SELECT TOP 30 i.InvoiceNumber, i.Branch AS InvBranch, m.Branch AS JobBranch
FROM ff.Invoice i LEFT JOIN ff.miscJob m ON m.JobID = i.JobId
WHERE i.Segment = 'M3' ORDER BY i.InvoiceDate DESC;                  -- B = 'to insert', rest fine
Solution

Applied and verified in the frontend on 23 Jul 2026 (invoice 248205 now shows BN260700001). Two parts: create Bangkok’s missing FY2026-27 FINV windows, and back-fill the first number of the new year onto the stranded invoice.

How to run this in SSMS — run the block ONCE, not statement by statement Highlight the entire block below and press Execute a single time. Running the INSERT, UPDATE and verify SELECTs separately caused confusion during this ticket (an INSERT ran alone, and a BEGIN TRAN left open held locks that briefly blanked the Invoice list on the frontend until COMMIT). This version has no open transaction — every statement auto-commits, so nothing is left pending. 
USE BSS;

-- Part A: create Bangkok's missing FY2026-27 FINV windows (fiscal Apr-Mar, YearCode 27).
--         N counter starts at 1 so it accounts for BN260700001 assigned below;
--         the next zero-GST Bangkok invoice will be ...00002. X and Z start at 0.
INSERT INTO appConfig.Numbers (BranchId, Segment, NType, Number, YearCode, DateFrom, DateTo, isActive, NFormat, POD)
VALUES
 ('B','N','FINV', 1, '27', '2026-04-01','2027-03-31', 1, 'Branch Taxtype YYMM99999',''),
 ('B','X','FINV', 0, '27', '2026-04-01','2027-03-31', 1, 'Branch Taxtype YYMM99999',''),
 ('B','Z','FINV', 0, '27', '2026-04-01','2027-03-31', 1, 'Branch Taxtype YYMM99999','');

-- Part B: back-fill the stranded invoice with the first N number of the new FY.
--         Format 'Branch Taxtype YYMM99999' -> B + N + 2607 + 00001 = BN260700001.
UPDATE ff.Invoice SET InvoiceNumber = 'BN260700001'
WHERE InvoiceId = 248205 AND InvoiceNumber = 'to insert';   -- 1 row affected

-- Verify
SELECT * FROM appConfig.Numbers WHERE BranchId='B' AND NType='FINV' AND YearCode='27';    -- 3 rows
SELECT InvoiceId, InvoiceNumber, Branch, Segment FROM ff.Invoice WHERE InvoiceId=248205;  -- BN260700001
Revert query — also run ONCE as a single block Safe only while no other Bangkok FINV invoice has drawn from these rows (i.e. the N row’s Number is still 1 and X / Z are still 0). Once other invoices issue real numbers from the new windows, do not delete the rows. 
USE BSS;

-- Put the invoice back to its original value
UPDATE ff.Invoice SET InvoiceNumber = 'to insert'
WHERE InvoiceId = 248205 AND InvoiceNumber = 'BN260700001';   -- 1 row affected

-- Remove the 3 FY2026-27 windows (guarded: only untouched rows)
DELETE FROM appConfig.Numbers
WHERE BranchId='B' AND NType='FINV' AND YearCode='27'
  AND DateFrom='2026-04-01' AND DateTo='2027-03-31'
  AND Segment IN ('N','X','Z')
  AND Number IN (0,1);                                        -- 3 rows affected

-- Verify
SELECT InvoiceId, InvoiceNumber FROM ff.Invoice WHERE InvoiceId=248205;                   -- 'to insert'
SELECT * FROM appConfig.Numbers WHERE BranchId='B' AND NType='FINV' AND YearCode='27';    -- 0 rows
Permanent fix (pending — code change) The data fix above resolves this invoice but the same failure will return at every fiscal-year boundary for any branch whose next window was not pre-created. appConfig.Get_Number should auto-create the next FY window and continue — it already does exactly this for NType = 'Q' (quotations) — instead of returning to insert. Alternatively a scheduled year-rollover job should clone every active series into the new FY before 1-April. Until that ships, series windows must be created manually each year (see Recurrence Risk).
Recurrence Risk
IssueRiskWill It Recur?Action Required Until Permanent Fix
Invoice number saved as “to insert” (lapsed FY window) High Yes — every appConfig.Numbers series carries a DateTo. Any branch / NType whose next fiscal-year (or calendar-year) window is not pre-created will strand the first invoice after the boundary with to insert. Bangkok’s X and Z segments were equally lapsed and are now fixed too, but other branches / NTypes may still lack their next window. Before each fiscal-year rollover, confirm a covering row exists for every active BranchId / NType / Segment. When any number saves as to insert, toinsert or ParameterErr, create/refresh the series row (correct DateFrom/DateTo/YearCode and the same NFormat as the prior window), then back-fill the number. Remember Bangkok is fiscal (Apr–Mar); Singapore/DXB are calendar (Jan–Dec).
Observations

Points that looked wrong during review but are expected behaviour — not bugs:

What was seenWhy it is not a bug
The “format is wrong” complaint — this invoice’s number differs from the other rows on the screen (BN… vs SGN…). The Invoice list shows 517 rows across branches. The SGN… rows are Singapore (code S → SG) invoices; the Bangkok format is genuinely BN…. Both are correct for their own branch. There is no separate format defect — the only fault was the missing FY window, which produced the literal to insert instead of any number.
Older Bangkok M3 invoices from 2023 were numbered SGN… (Singapore series) even though stored under branch B. That reflects earlier branch-derivation behaviour (the ff.Fn_BillBranch logic was changed on 03-Feb-2026). Current, correct behaviour numbers every M3 invoice under its own branch, as confirmed across all recent M3 rows.
The Invoice list briefly showed no rows on the frontend during the fix. An SSMS query window had an uncommitted BEGIN TRAN open, holding locks that blocked the site’s reads. Resolved by COMMIT. Always close a BEGIN TRAN block with COMMIT/ROLLBACK in the same session — or run these fixes without an explicit transaction, as documented above.
