Issue 1 — Add Email ID to Credit Application Alert Emails (West Africa & Dubai)

All West Africa branches & Dubai  |  Credit Application → Credit Application → Customer Credit Application

Key Takeaway Credit Application alert recipients are hardcoded inside masterOps.Get_Customer_Credit_Application_Mail, keyed by @Stage and the branch's CompanyId (1 = India, anything else = international). There is no email config table for this module, so any recipient change is a stored-procedure change. The SP only returns the To/Cc lists — BSSApi sends the mail (and only when called with isMail=1) — so executing the SP in SSMS is a safe way to preview recipients without sending anything.
Problem

Request from the business: add sreekumar.kurup@blackstoneshipping.com to the Credit Application Alert email list for all West Africa branches and Dubai, and confirm once done. Page: Credit Application → Credit Application → Customer Credit Application. It was not known at the time whether this required a procedure change or a database (data) change.

Investigation
Step 1 — Trace how the alert email is generated

Searched the frontend and BSSApi source for the Customer Credit Application mail flow. When a stage is confirmed on the screen, the Angular component calls: 

GET /CustomerControllerApplication/Get_Customer_Credit_Application_Mail?TransID=..&Stage=..&isMail=1
(BLISS/src/app/blisscredit-control/customer-credit-application*.component.ts)

The API endpoint (BSSApi/Controllers/FIN_CustomerCreditApplication_Controller.cs) executes masterOps.Get_Customer_Credit_Application_Mail and, when isMail=1 and the SP returns isMail=1, sends the email itself via SMTP using the mTo / mCc / mBcc columns from the SP result.

Recipient lists are built entirely inside the SP — hardcoded strings selected by @Stage and @CmpID (OrganizationUnit.CompanyId of the application's branch). No email configuration table is involved.

Conclusion: this is a stored-procedure change, not a data update.

Step 2 — Identify the branch codes for West Africa and Dubai

Listed all branches and checked which codes actually file credit applications:

select OuID, EdiCode, OuName, CompanyId, isActive
from masterCom.OrganizationUnit
order by CompanyId, OuName;

select Branch, count(*) Apps, max(ADate) LastApp
from masterOps.Customer_Credit_Application
group by Branch order by Branch;
DXB = Dubai (CompanyId 10, 198 apps)  |  GHA = Ghana (CompanyId 12, 47 apps) LFW = Lome (CompanyId 13, 16 apps)  |  ABJ = Abidjan (CompanyId 14, 22 apps)

Conclusion: the target branches are DXB, GHA, LFW, ABJ — all with CompanyId <> 1, so they all flow through the SP's international recipient blocks.

Step 3 — Review the live SP recipient logic

Pulled the live SP text to confirm the recipient blocks:

exec sp_helptext 'masterOps.Get_Customer_Credit_Application_Mail';
To = salesperson + Incentive.SalesUser reporting chain + branch head (OfficialEMail from masterUser.UserMaster). Then, for CompanyId <> 1 branches: Stage 1 "Credit Application Submitted" -> Cc = g.pravin; credit.global; ar.intl Stage 3 "Credit Analysis Done" -> To = g.pravin; credit.global; ar.intl; + branch head Stage 4 "Credit Decision Made" -> Cc = g.pravin; credit.global; ar.intl Stages 2 and 5 never set isMail=1, so no email is ever sent for them.

Conclusion: three alert mails exist per application (Submitted / Analysis Done / Decision Made), and the West Africa + Dubai branches share the same hardcoded lists as every other non-India branch. A branch-scoped addition needs a condition on @Branch, which the SP already holds.

Step 4 — Baseline: capture current recipients (no mail is sent by the SP)

Executed the SP for one recent Dubai and one recent Abidjan application to record the before state:

exec masterOps.Get_Customer_Credit_Application_Mail @TransID = 3840, @Stage = 1;  -- DXB
exec masterOps.Get_Customer_Credit_Application_Mail @TransID = 3838, @Stage = 1;  -- ABJ
DXB (3840): isMail=1 mTo = r.rajesh@blackstoneshipping.com;b.munavir@blackstoneshipping.com mCc = g.pravin@blackstoneshipping.com;credit.global@blackstoneshipping.com;ar.intl@blackstoneshipping.com ABJ (3838): isMail=1 mTo = it@blackstoneshipping.com;n.rajesh@blackstoneshipping.com;r.chandra@blackstoneshipping.com mCc = g.pravin@blackstoneshipping.com;credit.global@blackstoneshipping.com;ar.intl@blackstoneshipping.com

Conclusion: confirmed the hardcoded Cc trio on both branches and that sreekumar.kurup is nowhere in the lists. Baseline recorded for before/after comparison.

Root Cause
masterOps.Get_Customer_Credit_Application_Mail — recipients hardcoded per @Stage + CompanyId The Credit Application Alert recipient lists are literal strings inside the stored procedure, selected only by stage and by whether the branch belongs to the India company (CompanyId = 1) or not. There is no branch-level email list anywhere in the database, so a recipient cannot be added for specific branches without altering the procedure.
Solution

Alter masterOps.Get_Customer_Credit_Application_Mail (BSS database) and insert the block below after the last if @Stage=4 and @CmpID<>1 … End block, immediately before the final select @lHtml Html, @isMail isMail, … line (SSMS → right-click the SP → Script Stored Procedure as → ALTER To): 

-- Task 31286: add Sreekumar Kurup to Credit Application alerts for West Africa + Dubai
if @Branch in ('DXB','GHA','LFW','ABJ')
    set @mCc = case when isNull(@mCc,'')='' then 'sreekumar.kurup@blackstoneshipping.com'
                else @mCc + ';sreekumar.kurup@blackstoneshipping.com' end

This adds the address as Cc on all three alert mails (Stage 1 Submitted, Stage 3 Analysis Done, Stage 4 Decision Made) for the four branches only. Every other branch and all India mails are untouched. Stages 2 and 5 never send mail, so the extra Cc there is inert.

Verification — re-run the baseline calls after the ALTER; mCc must now end with the new address, and a non-target branch (e.g. a Rotterdam TransID) must return an unchanged mCc:

exec masterOps.Get_Customer_Credit_Application_Mail @TransID = 3840, @Stage = 1;  -- DXB: mCc ends ;sreekumar.kurup@...
exec masterOps.Get_Customer_Credit_Application_Mail @TransID = 3838, @Stage = 1;  -- ABJ: mCc ends ;sreekumar.kurup@...

Revert — script the SP as ALTER again, delete the three inserted lines, execute.

Scope note The request said "Credit Application Alert email list", which was read as all three alert mails of the module. If the requester meant only the Stage-1 "Credit Application Submitted" alert, change the condition to if @Stage=1 and @Branch in ('DXB','GHA','LFW','ABJ'). Live-mail confirmation happens naturally on the next credit application submitted from any of the four branches.
Recurrence Risk
IssueRiskWill It Recur?Action Required Until Permanent Fix
Recipient lists hardcoded in the SP Medium Yes — every future add/remove of a Credit Application alert recipient requires another ALTER of this SP. The lists are duplicated across six stage/company blocks (and again in the unused sibling SP …_Mail_By_LeadID), so partial edits are easy to get wrong. Use this report as the map of where each list lives. Permanent improvement: move recipients to a config table keyed by branch/stage (same pattern as ff.Email used by Cheque Request).
Address-list truncation Low Possible — @mTo / @mCc are varchar(500). A long sales reporting chain plus appended addresses could silently truncate the list (SQL Server truncates on assignment without error here). When adding more recipients in future, check the combined length of the longest To/Cc string.
