Issue 1 — Cheque Request re-send "not going through" / email not sent (Singapore)

Job 2602711  |  Branch: Singapore (S)  |  QA context: Job 2608785, Vendor Invoice 8506988862

Key Takeaway When a Cheque Request behaves strangely for a whole branch (not one job), check the Purchase Entry No first. appConfig.Get_Number returns the literal string ParameterErr when no appConfig.Numbers series exists for the branch, and ff.Update_2ChequeRequest saves it without any validation. Also remember: Get_Number remaps login branch codes before the lookup (S→SG, Y→NY, W→ANT, R→RTM, E→UKM), so the series row must exist under the remapped code.
Problem

Ops (Singapore) booked a DN & credit note and sent the cheque request on 19.03.2026. Finance rejected it, and when Ops tried to re-send the cheque request through BLISS it "did not go through", blocking file closing (Job 2602711). QA added that for Job 2608785 / vendor invoice 8506988862 the finance team rejected the purchase posting, Ops re-sent the cheque request, but the cheque-request mail was not sent again. The developer confirmed that in the data the request showed as sent to finance for all charges — yet no email arrived.

Investigation
Step 1 — Traced the send and mail flow in code

The Cheque Request page (BLISS > bilssfinance-operation > cheque-request) posts to /ChequeRequest/Update_2ChequeRequest → SP ff.Update_2ChequeRequest. The SP generates a Purchase Entry No per vendor invoice via appConfig.Get_Number @Branch, '', 'VINV', stamps ChqReqSent / PurEntryNo on the cost rows, and returns the mail body (Table1) and recipients (Table2, from ff.Email where EType='CHQREQUEST'). The email itself is then sent by the browser via /mail/sendChequeRequestMail — a DB update can succeed while the mail silently fails. Rejection goes through /tallyMigration/Update_ChqReq_Reject → SP ff.Update_ChqReq_Reject, which clears ChqReqBy / ChqReqSent / PurEntryNo / PurEntryDate on all cost rows matching the rejected PurEntryNo.

Step 2 — Checked the job data: the failing items are Import rows with a corrupted Purchase Entry No

The export job 2602711 turned out to belong to branch GHA and its export cheque requests were healthy (proper VINVGHA26xxxx numbers). The failing items were the Import booking rows under Singapore. The send audit trail showed the smoking gun: 

select Typ, JobNumber, VendorInvNo, PurEntryNo, Branch, CreatedBy, Created
from Bss_Report.ff.Cheque_Request_List
where JobNumber in ('2602711','2608785') or VendorInvNo = '8506988862'
order by Created desc;
Imp 2607794 8506988862 ParameterErr S ... 2026-07-08 06:12:59 Imp 2602711 2553204170 ParameterErr S ... 2026-07-01 19:34:05 Imp 2602711 2553206870 ParameterErr S ... 2026-07-01 19:34:05 Imp 2602711 (5 invoices) ParameterErr S ... 2026-03-19 12:41:42 Exp 2602711 ... VINVGHA26xxxx GHA ... (export side healthy)

Every Singapore import cheque request — including the original 19.03.2026 send and all re-sends — was saved with PurEntryNo = 'ParameterErr'. (Vendor invoice 8506988862 actually belongs to job 2607794, not 2608785.) The re-sends did run and produced mail-body rows, so the failure was not in the send itself.

Step 3 — Read appConfig.Get_Number: "ParameterErr" means no number series configured

appConfig.Get_Number (BSS) returns the literal string 'Parameter Err' (spaces stripped → ParameterErr) when no row exists in appConfig.Numbers for the branch + NType + Segment. ff.Update_2ChequeRequest saves this value straight into PurEntryNo with no validation. Critically, the BSS version of the SP also remaps the login branch code before the lookup: 

if @Branch='W' set @Branch='ANT'
if @Branch='R' set @Branch='RTM'
if @Branch in ('E','LON') set @Branch='UKM'
if @Branch='S' set @Branch='SG'
if @Branch='Y' set @Branch='NY'

So Singapore requests look up BranchId='SG' — a fix inserted under BranchId='S' would do nothing.

Step 4 — Confirmed no VINV series exists for the remapped codes
select * from appConfig.Numbers where BranchId in ('SG','NY','D') and NType = 'VINV';
(0 rows)

The full VINV listing showed active series for GHA, T, M, RTM, ANT, UKM, HAM, DAC, etc. — but nothing for SG, NY or D. The number generator's own call log proved which codes actually request VINV numbers (each call is logged twice — pre- and post-remap — hence the identical pairs): 

select Branch, Segment, count(*) as calls
from ffLog.Lg_AppConfig_Number
where NType = 'VINV'
group by Branch, Segment
order by calls desc;
W 29733 = ANT 29733 (works: ANT series exists) R 18894 = RTM 18894 (works: RTM series exists) E 9466 = UKM 9466 (works: UKM series exists) Y 7013 = NY 7013 (FAILS: no NY series) S 1219 = SG 1219 (FAILS: no SG series) D 1272 (not remapped) (FAILS: no D series)

GHA and LFW had this exact failure until appConfig.Numbers rows were created for them in early 2025 — the same one-row fix proposed here, with precedent.

Step 6 — Verified the mail generation and recipients were intact

On every re-send the SP produced mail-body rows (proven by Bss_Report.ff.Cheque_Request_List inserts) and the recipient lookup resolved: branch 'S' is in the SP's BizControl-routing list, the bookings' controlling office is GHA (OuID 52), and ff.Email has an active CHQREQUEST row for GHA. So the mail was generated and addressed to the West-Africa distribution (see Observations) — the perceived "email not sent" and "request not going through" are the effects of Issue 2 below: the request kept getting silently un-sent after every rejection anywhere in the branch.

Root Cause
appConfig.Numbers — no VINV series for BranchId 'SG' (Singapore's remapped code) appConfig.Get_Number remaps login branch 'S' to 'SG' and finds no NType='VINV' series, so it returns the literal ParameterErr, which ff.Update_2ChequeRequest saves as the Purchase Entry No for every Singapore cheque request (2,062 occurrences since Jul 2023). Branches Y (remapped to 'NY', 11,661 occurrences) and D (not remapped, 3,823 occurrences) have the identical defect.
Solution

Prepared 08 Jul 2026 — to be executed and then verified in the frontend. The fix for this task is the SG series row; NY and D are the same defect and should be applied after confirming with the finance team (see the note below). 

-- STEP A (pre-flight): confirm SG's year-window convention from its existing series
select NType, Segment, YearCode, DateFrom, DateTo, NFormat
from appConfig.Numbers
where BranchId = 'SG' and isActive = 1 and getdate() between DateFrom and DateTo
order by NType;
-- Expect calendar-year windows (Jan-Dec), like GHA/LFW/RTM/ANT.
-- If SG's rows are fiscal (Apr-Mar), use DateFrom '2026-04-01', DateTo '2027-03-31', YearCode '27' below instead.

-- STEP B (preview): expect 0 rows
select * from appConfig.Numbers where BranchId = 'SG' and NType = 'VINV';

-- STEP C (fix, 1 row): first number issued will be VINVSG260001
insert into appConfig.Numbers (BranchId, Segment, NType, Number, YearCode, DateFrom, DateTo, isActive, NFormat, POD)
values ('SG', '', 'VINV', 0, '26', '2026-01-01', '2026-12-31', 1, 'TypeBranchFY9999', '');

-- STEP D (verify): expect 1 row, Number = 0
select * from appConfig.Numbers where BranchId = 'SG' and NType = 'VINV';

-- REVERT (only while Number is still 0, i.e. before any number has been issued):
-- delete from appConfig.Numbers where BranchId = 'SG' and NType = 'VINV' and Number = 0;

Issue 2 — Rejecting one "ParameterErr" entry un-sends every pending request sharing the key

All branches without a VINV series  |  SP: ff.Update_ChqReq_Reject

Key Takeaway ff.Update_ChqReq_Reject clears cheque-request state with WHERE PurEntryNo = @PurEntryNo. That is safe only while Purchase Entry Numbers are unique. The moment a branch writes a shared value (ParameterErr), one rejection wipes the entire branch's pending queue — and the affected users only see their requests silently vanish.
Problem

This is the mechanism behind "we re-send it and it never goes through". Ops re-sent their cheque request, the data showed it as sent, and some time later it was back to not-sent without anyone rejecting their invoice.

Investigation
Step 1 — Reject log shows mass wipes on every "ParameterErr" rejection
select top 50 * from Bss_Log.ffL.ChqReq_Reject_Log order by Rejected desc;
TransID 724 2026-07-08 06:47 PurEntryNo=ParameterErr → cleared 4 cost rows TransID 723 2026-07-08 06:41 PurEntryNo=ParameterErr → cleared 40 cost rows (incl. invoice 8506988862's row 340211) TransID 722 2026-07-07 11:41 PurEntryNo=ParameterErr → cleared 13 cost rows TransID 721 2026-06-16 01:46 PurEntryNo=ParameterErr → cleared 150 cost rows TransID 720, 703, 682, 681 ... each ParameterErr reject cleared 15-59 rows (normal rejects, e.g. VINVRTM265544, cleared exactly the 1-4 rows of their own invoice)

Each "ParameterErr" rejection nulls ChqReqBy / ChqReqSent / PurEntryNo / PurEntryDate on every cost row in the system that carries the shared key at that moment — across unrelated jobs and vendors. Rejections of properly-numbered entries only touch their own invoice.

Step 2 — Timeline for the reported invoice

Invoice 8506988862 (job 2607794): re-sent 08 Jul 06:12 → wiped at 06:41 by a rejection of a different invoice sharing the key (TransID 723). Job 2602711's import invoices: sent 19 Mar, re-sent 29 Jun and 01 Jul — each send later found cleared again. The finance approval page (ff.Get_2ChequeRequest_Approve) additionally groups by PurEntryNo, so all pending Singapore items appear lumped under one "ParameterErr" entry — rejecting what looks like one invoice rejects everything.

Root Cause
ff.Update_ChqReq_Reject — keys on PurEntryNo, which is not unique when number generation fails The reject SP clears cheque-request state for all rows matching the rejected PurEntryNo. Combined with Issue 1 (all Singapore requests sharing ParameterErr), every rejection mass-wipes the branch's pending queue, creating the endless send→vanish loop that was reported as "the cheque request is not going through".
Solution

No separate data fix — Issue 1's series row stops new shared keys, after which rejections become precise again. Until the series rows exist for all affected branches (NY, D):

Interim caution for finance Do NOT reject any purchase-posting entry whose Purchase Entry No shows ParameterErr — each such rejection un-sends every other pending request in that branch. Ask IT to fix the branch's number series first, have Ops re-send, then reject the properly-numbered entry.
Permanent fix pending Change ff.Update_ChqReq_Reject to clear rows by the CostID list of the rejected entry (the SP already computes @CostIDList for its log) instead of WHERE PurEntryNo=@PurEntryNo.
