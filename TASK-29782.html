Issue 1 — Ship Transport report: missing / abnormal Distance & Emission

Jobs 2504482, 2504483, 2504486, 2504498, 2501739, 2501529  |  Sustainability > Ship Transport  |  Year 2025

Key Takeaway On the Ship Transport report, Distance and Emission (MTCO2e) are pre-stored on the job row (ff.BJob, summed up from ff.BBL) — the report only displays them, it does not compute them. A blank or zero value therefore means the value was never stamped, and the stamping depends on a port-to-port distance existing in ff.PortMaster_Distances. That distance is fetched from an external route API that is coordinate-driven: if a port has no LatLon, no distance can ever be produced for any lane that uses it — and with no distance, emission stays 0.
Problem

On Sustainability > Ship Transport for Year 2025, several jobs show missing or abnormal data:

  • Distance not showing — Jobs 2504482, 2504483
  • Emission not showing — Jobs 2504486, 2504498
  • Abnormal value showing (reported as transit days) — Jobs 2501739, 2501529

Not a billing issue — this is a reporting / emissions feature, so no invoices are involved. Branch was not specified at reporting time and had to be verified per job. Reported endpoint was expected to be the Sustainability report SP, to be confirmed during investigation.

Investigation
Step 1 — Identify the report SP and classify all six jobs

The Ship Transport grid is served by ReportController/Rpt_Sustainability_01 (params Branch, Date1, Date2, DType). Reading the SP confirms the export grid displays j.Distance and j.MTCO2e straight off ff.BJob — no live calculation. First we looked up the six jobs and their stored values. 

select JobNumber, Branch, Segment, JobDate, POLID, PODID,
       GrossWt, T20, T40, Distance, MTCO2e
from ff.BJob
where JobNumber in ('2504482','2504483','2504486','2504498','2501739','2501529');
Job Br Seg T20 T40 Distance MTCO2e 2504482 O E1 0 1 NULL 1.01506 ← distance blank, emission OK 2504483 O E1 0 1 NULL 1.66484 ← distance blank, emission OK 2504486 C E2 2 0 NULL 0.00000 ← both blank 2504498 C E2 0 2 NULL 0.00000 ← both blank 2501739 GHA E1 20 0 97060 0.00000 ← distance present, emission 0 2501529 S E2 10 0 110920 0.00000 ← distance present, emission 0

All six are export jobs (ff.BJob). The symptoms split into three clearly different states — already a sign of more than one root cause. Note POLID/PODID are NULL on ff.BJob itself; the report reads POL/POD from the booking, so that is not the display bug.

Step 2 — Replicate the SP's distance & emission-factor lookup per job

We rebuilt the SP's exact join chain (booking POL/POD → PortMaster_Distances → TradeLane_Sustain → CountryRegion_Sustain) and called ff.fn_sustain to see what emission would be produced today.

Job POL → POD PortPairKm Factor fn_sustain 2504482 Mundra → Rotterdam 6313 / 10895 79.8 1.111 (2 conflicting rows) 2504483 Mundra → Rotterdam 6313 / 10895 79.8 1.833 2504498 Ennore → Cartagena 9575 66.4 2.708 (all inputs present!) 2504486 Ennore → Algiers NULL 66.4 0 (no distance row) 2501739 Tema → Baltimore NULL 166.1 0 (no distance row) 2501529 Lome → Wuhan Pt NULL 105.7 0 (no distance row)

The emission factor resolves for every job. The differentiator is PortPairKm: three lanes have no PortMaster_Distances row at all. fn_sustain shows 2504498 could be computed right now (2.708) — it was simply never stamped.

Step 3 — Confirm the three lanes are absent from the distance table
select FromCode, ToCode, Distance
from ff.PortMaster_Distances
where (FromCode in ('INENR','GHTEM','TGLFW') and ToCode in ('DZALG','USBAL','CNWHG'))
   or (FromCode in ('DZALG','USBAL','CNWHG') and ToCode in ('INENR','GHTEM','TGLFW'));
(0 rows) — Ennore→Algiers, Tema→Baltimore and Lome→Wuhan have no distance row in either direction.

The bss_audit.ff.Sustainability_Port_Error log carries repeated "Port pair Not Found!" entries for Ennore→Algiers (last tried 17 Jul 2026) and Tema→Baltimore (last tried 21 Apr 2026). Lome→Wuhan has no error rows at all — the fetch was never even dispatched for it.

Step 4 — Trace the values to BL level (how the report totals are built)

BJob.Distance/MTCO2e are a SUM() over the job's BL-containers of ff.BBL.Distance/MTCO2e. The per-container sum also explains the "abnormal" figure.

Job BLID BJob_Dist BL_Dist BL_MTCO2e ContCount 2501529 58272 110920 11092 0 10 → 11092 × 10 = 110920 2501739 58418 97060 4853 0 20 → 4853 × 20 = 97060 2504482 60620 NULL NULL 1.01506 1 2504486 60624 NULL NULL 0 2 2504498 60635 NULL NULL 0 2

The reported "abnormal transit days" is really the total container-distance (per-container distance × container count). There is no transit-days column in this report; the large number is Distance, inflated by the container count. Also visible here: jobs 2504482/83 carry a valid BL.MTCO2e but a NULL BL.Distance.

Step 5 — Why the distance fetch fails: missing port coordinates

The distance is fetched by ff.Get_Port_Distances_API, which passes each port's LatLon to an external route API. If a port has no coordinates, the call cannot route and logs "Port pair Not Found!". We checked the six ports involved: 

select PortId, PortCode, PortName, LatLon
from masterOps.Portmaster
where PortId in (61043,29983,56490,90602,87035,16720);
INENR Ennore 80.352465,13.253365 ← has coords GHTEM Tema 0.012948,5.629084 ← has coords DZALG Algiers NULL ← MISSING USBAL Baltimore NULL ← MISSING TGLFW Lome NULL ← MISSING CNWHG Wuhan Pt NULL ← MISSING

Every blocked lane has at least one coordinate-less port (Lome→Wuhan has two). A check of the staging copy (Bss_Dev.ff.PortMaster_Distances) confirmed these lanes were never captured there either. This — not a vendor outage — is why the distances were never produced.

Step 6 — The "distance blank" symptom is systemic (code defect)

Jobs 2504482/83 have emission but no distance because the export BL-save (Update_BBL) captures only MTCO2 from ff.fn_Invoice_MTCO2_Gen and discards the Distance it returns — and no other code writes ff.BBL.Distance. We measured how widespread this is across the 2025 export book: 

-- emission present but distance blank
select count(*) from ff.BJob
where JobDate >= '2025-01-01' and JobDate < '2026-01-01'
  and Segment like 'E%' and Segment not in ('E0','E8')
  and isNull(MTCO2e,0) > 0 and isNull(Distance,0) = 0;   -- 9253

-- both populated (healthy)
select count(*) ... and isNull(Distance,0) > 0;           -- 1248
9,253 export jobs with emission but NO distance vs 1,248 healthy → ~88% of emission-bearing 2025 export jobs show a blank Distance.

"Distance not showing" is therefore not specific to the two reported jobs — it is close to the default state of the export Ship Transport report, driven by the Update_BBL defect.

Root Cause
Three distinct causes behind the six jobs ① Missing port coordinates — masterOps.Portmaster.LatLon is NULL for Algiers (29983), Baltimore (90602), Lome (87035), Wuhan (16720). The distance fetch (ff.Get_Port_Distances_API) is coordinate-driven, so lanes using these ports never get a PortMaster_Distances row → no distance and no emission. Affects: 2504486, 2501739, 2501529 (emission 0; 2501739/2501529 carry a legacy distance from an older path).
② Export BL-save discards Distance (code defect) ff.Update_BBL runs select @MTCO2e = MTCO2 from ff.fn_Invoice_MTCO2_Gen('BL',@BLID,'E1') but ignores the Distance column the function also returns, and nothing else writes ff.BBL.Distance. Result: export jobs get an emission with a blank distance. Systemic — ~9,253 / ~10,500 (88%) of 2025 emission-bearing export jobs are affected. Affects (reported): 2504482, 2504483.
③ Never re-stamped 2504498 (Ennore→Cartagena) has a distance row (9575), a resolving factor, and fn_sustain = 2.708 — but the BL emission/distance were never computed. A re-stamp fills it.
Solution
Status: PROPOSED — not yet applied, not yet verified in the frontend. Nothing below has been run against the database. Coordinates in Part A are approximate and must be verified against an authoritative port source before applying. Steps to run in order, each with its own revert.

Part A — seed the four missing port coordinates (format is "Lon,Lat", west/south negative).

-- Preview (expect 4 rows, all LatLon NULL)
select PortId, PortCode, PortName, LatLon
from masterOps.Portmaster
where PortId in (29983,90602,87035,16720);

-- Apply (expect 4 rows affected)
update masterOps.Portmaster set LatLon='3.068,36.768'   where PortId=29983 and LatLon is null;  -- Algiers
update masterOps.Portmaster set LatLon='-76.535,39.267' where PortId=90602 and LatLon is null;  -- Baltimore
update masterOps.Portmaster set LatLon='1.284,6.128'    where PortId=87035 and LatLon is null;  -- Lome
update masterOps.Portmaster set LatLon='114.305,30.593' where PortId=16720 and LatLon is null;  -- Wuhan

-- Revert
update masterOps.Portmaster set LatLon=NULL where PortId in (29983,90602,87035,16720);

Part B — fetch the three lane distances. Side-effecting: makes a live external call to the route API and inserts a cache row per lane.

exec ff.Get_Port_Distances_API 61043, 29983, 'S', '';   -- Ennore  -> Algiers
exec ff.Get_Port_Distances_API 56490, 90602, 'S', '';   -- Tema    -> Baltimore
exec ff.Get_Port_Distances_API 87035, 16720, 'S', '';   -- Lome    -> Wuhan

-- Verify the new rows exist
select FromCode, ToCode, Distance, Created from ff.PortMaster_Distances
where (FromCode in ('INENR','GHTEM','TGLFW') and ToCode in ('DZALG','USBAL','CNWHG'))
   or (FromCode in ('DZALG','USBAL','CNWHG') and ToCode in ('INENR','GHTEM','TGLFW'));

-- Revert (delete ONLY the rows created by this fix — check Created / CreatedBy first)
delete from ff.PortMaster_Distances
where (FromCode in ('INENR','GHTEM','TGLFW') and ToCode in ('DZALG','USBAL','CNWHG'))
   or (FromCode in ('DZALG','USBAL','CNWHG') and ToCode in ('INENR','GHTEM','TGLFW'));

Part C — re-stamp the affected BLs / jobs. First capture the current values (this snapshot is the revert source), then re-stamp. Exact per-job stamp values are finalised once Part B returns the fetched distances.

-- Baseline snapshot (revert source) — keep these tables until the fix is signed off

into  bss_audit.dbo.Revert_29782_BJob

-- Re-stamp: preferred = re-save each booking in the app (runs the full correct chain);
--           alternative = controlled per-job UPDATE of BBL.Distance/MTCO2e then re-sum to BJob,
--           using the distances fetched in Part B. Finalised after Part B.

-- Revert (restore originals)
update b set b.Distance=r.Distance, b.MTCO2e=r.MTCO2e

from ff.BJob j join bss_audit.dbo.Revert_29782_BJob r on r.JobID=j.JobID;
